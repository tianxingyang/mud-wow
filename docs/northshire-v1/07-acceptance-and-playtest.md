# 07：验收与真人试玩

> 状态：首版验收基线
> 范围：北郡 1–5 级垂直切片
> 依赖：[00-slice-contract.md](./00-slice-contract.md)、[01-player-flow.md](./01-player-flow.md)、[02-world-and-content.md](./02-world-and-content.md)、[03-quest-state-pack.md](./03-quest-state-pack.md)、[04-combat-and-progression.md](./04-combat-and-progression.md)、[05-tactics-and-decisions.md](./05-tactics-and-decisions.md)、[06-multiplayer-and-recovery.md](./06-multiplayer-and-recovery.md)

本文只定义“如何证明北郡 V1 成立”，不替代 00–06 作玩法决定。依赖规则尚未冻结时，相应用例标记为 `BLOCKED`，不得猜测预期结果。

## 1. 验收结论

北郡 V1 必须同时通过四层证据：

1. **自动化测试：**证明纯规则、状态机和内容引用在确定输入下正确。
2. **集成测试：**证明协议、数据库事务、多人并发和恢复边界正确。
3. **E2E 测试：**证明真实客户端到权威服务端的完整路径可以跑通。
4. **真人试玩：**证明玩家看得懂、走得通、能作出有价值的战斗选择，并能在 60–90 分钟完成切片。

测试状态统一使用：

| 状态 | 含义 |
|---|---|
| `PASS` | 步骤执行完毕，证据满足阈值 |
| `FAIL` | 步骤执行完毕，结果未达到阈值 |
| `BLOCKED` | 依赖规则、内容或环境未冻结，无法得出唯一结论 |
| `NOT_RUN` | 可以执行，但本轮尚未执行 |

`BLOCKED` 和 `NOT_RUN` 都不计为通过。

当前 00–06 的 v1 规则均已冻结，初始验收计划不得把下述边界标为 `BLOCKED`；该状态只保留给未来确实重新打开的规则、缺失环境或新增依赖。

## 3. 固定测试条件

每次正式验收记录以下信息：

- 代码提交或构建 ID；
- 内容版本与数据库迁移版本；
- 客户端版本；
- 测试环境和浏览器版本；
- 随机种子、可控时钟配置和基准角色配置；
- 参与人数及每人的 Vanilla、MUD 和本构建熟悉度；
- 是否允许内置 `help`；
- 是否发生主持人提示、管理员修复或数据重置。

真人试玩使用当次冻结的标准初始角色、装备和默认战术。首次开战前，客户端或内置帮助必须主动暴露一次 `tactics` 入口，测试者至少查看一次个人战术；允许保持模板默认，不强制为了测试而修改。该步骤只验证入口可发现，不建立服务端解锁门槛；另以未查看战术的基准角色验证一次合法 `attack`，不得因此被拒绝。参与者不得阅读外部地图、路线攻略或任务答案；内置帮助可以使用，但必须计数。主持人不得主动提示，只有测试者明确求助或出现确定的软件阻塞时才介入并记录原因。

## 4. 最小遥测与记录

验收环境至少记录以下结构化事件或等价证据：

- 会话开始、角色进入、断线、重连和会话结束；
- `look`、查看出口、移动、与 NPC 交互及内置帮助使用；
- 命令被接受或拒绝、稳定错误码及是否造成状态变更；
- 任务接取、目标推进、交付、奖励和后续解锁；
- 战斗开始、覆盖动作、关键意图、胜负、时长、生命和资源变化；
- 经验、等级、物品、装备和训练状态变化；
- 奖励事务 ID、命令幂等 ID 和奖励唯一键；
- 房间成员、战斗参与者、最终快照中的 `scopeCursors[] = { scope, scopeEpoch, serverSeq }[]`、控制连接的 `controlEpoch`；
- 主持人提示、管理员干预、软锁、崩溃和数据修复。

聊天正文和敏感凭据不进入普通验收日志。真人试玩报告可以引用匿名参与者编号，不记录不必要的个人信息。

## 5. 自动化验收

### 5.1 内容与任务

- 所有发布内容通过 Schema、ID 唯一性、引用存在性和范围校验；错误数必须为 **0**。房间、NPC、敌人模板、出生点、任务、Harvest Node、普通／任务物品和技能都必须具有 02 §8.1 的通用来源字段，`source_version` 必须为 `vanilla_1_12`，`canonical_status` 只能是 `original | adapted | new`。
- 任一混合原版事实与 v1 改写的记录必须整体标为 `adapted` 并逐项列出 `adaptation_notes[]`；误用 `canonical`、混合记录标成 `original` 或缺少说明的记录数必须为 **0**。
- 同一构建全部内容记录的 `content_version` 必须是一个完全相同的包级不可变值；混合版本、逐记录私自版本或内容已改变但仍复用旧包版本的记录数必须为 **0**。
- 发布清单必须恰好包含 11 个房间、10 条任务、6 类敌人模板、20 个固定敌人出生点、8 个个人 Harvest Node、7 名北郡功能 NPC 和 1 个不可生成的外部 NPC 引用 `npc_marshal_dughan`；还必须恰好包含 04 第 5 节的 12 个 ability，以及 04 第 10.4／10.6 节和 03 §4.1 合计的 27 个 item。数量、精确 ID 集合或 ability 替换关系偏差必须为 **0**。
- 10 条任务的 `objectives[]` 必须恰好使用 03 第 3 节冻结的 10 个 `objectiveId`，全局唯一、类型一致且引用存在；缺失、改名、重复或以数组序号代替 ID 的记录数必须为 **0**。
- 10 条任务的 `giver_npc_id`、`turn_in_npc_id` 和 `prerequisite_quest_ids` 必须逐行等于 03 第 3 节的稳定 ID；所有 `knowledge_requirements=[]`、`repeatable=false`。人名、“同 NPC”或原任务号被写入这些 Schema 字段的记录数必须为 **0**。
- 目标的类型化 NPC／mob／item／node 引用、`required_count` 及两个稳定 `topic_id` 必须与 03 第 3／8 节精确一致；NS11 必须恰好包含一个内联 `exit_ns11_south_slice_complete`，独立顶层 trigger、`encounter` 和 `zone` 记录数都必须为 **0**。
- `npc_marshal_dughan` 只允许作为 #54 的切片外声明式 `turn_in_npc_id`／`target_npc_id` 引用；它出现在北郡运行时房间、spawn、可见实体、occupants 或 `currentlyInteractable` 集合中的次数必须为 **0**。
- 房间图从切片起点到所有必达目标及切片出口均可达；非预期死路和非预期循环必须为 **0**。
- 冻结的主任务链按前置顺序可完成；跳过前置、重复接取、重复交付和伪造目标均不能推进权威状态。
- 九条可在切片内交付的任务各至少有一条成功交付路径；#54 使用 03 §9.2 的跨切片专项路径。每种目标类型至少有一条失败或非法输入测试。
- 任务奖励、后续解锁和知识状态在相同输入下重复运行 **100 次**，结果必须完全一致。
- #54 的接取必须原子写入 `active`、一份 Documents 与 `assigned_to_goldshire`；#3905 的接取必须原子写入一份 Manifest 并直接进入 `ready`。接取失败、重复接取、放弃后重接和响应丢失均不得产生多份物品或半份状态。
- #3904 采集 1–8 个节点后放弃再重接时必须生成新的 `questRunId`；旧进行期的幂等记录保留但不阻止新进行期重新采集，每个新 `(characterId, questRunId, nodeId)` 仍只能成功一次。
- Bandana 和 Garrick's Head 必须在对应敌人结算事务中直接进入任务物品栏并同步推进目标；普通背包已满、未执行 `loot` 或执行 `decline-loot` 都不得阻止或删除任务物品。
- #54 到达南路边界后仍为 `active`，本地 `ready`、`completed`、`quest.turn_in` 和任务奖励发生次数必须均为 **0**；完成前后放弃／重接及重连行为必须逐项通过 03 §9.2。
- 固定任务装备奖励遇到普通背包满时，`quest.turn_in` 必须以 `QUEST_REWARD_INVENTORY_FULL` 整体拒绝，任务保持 `ready`，任务物品、XP、铜币、知识和解锁变化均为 **0**；旧 `commandId` 返回原拒绝，腾出空间后用新 `commandId` 只能成功一次。

### 5.2 命令与状态安全

- 未知命令类型、坏 `protocolVersion`、畸形 envelope 和类型 payload 缺失参数必须返回导入协议的稳定 `ProtocolErrorResult`，不带 `firstResult`、不占用 `commandId`、不进入持久幂等或领域状态；非法目标、当前状态不允许和重复的已注册命令则按普通 `CommandResult` 拒绝。所有场景未声明状态变化必须为 **0**。
- 对同一有状态命令重复提交相同幂等 ID **1,000 次**，永久状态只能变化一次。
- 同一 `(actorId, commandId)` 携带不同的有效命令类型或规范化载荷时，必须稳定返回无 `firstResult/delivery`、`retryable = false` 的 `ProtocolErrorResult(COMMAND_ID_REUSE_CONFLICT)`；既有首次幂等记录和领域状态不得改变或泄漏。之后重发原始完整 envelope 仍必须取得首次普通 `CommandResult`，改正后的新业务意图只能使用新 `commandId`。
- 网络重试必须复用原始完整 command envelope，包括原 `clientSeq`；命中 `(actorId, commandId)` 后即使该序号已落后也返回首次结果，只有新 `commandId` 才执行 `clientSeq` 新鲜度校验。
- 角色创建等绑定前命令必须由服务端派生 `account:{accountId}`，角色命令必须派生 `character:{characterId}`；二者跨重连分别命中原幂等结果，且相同 `commandId` 位于不同 actor 命名空间时互不碰撞。任何以 `sessionId`、`connectionId` 或客户端载荷作为 `actorId` 的次数必须为 **0**。
- 命令首次结果属于旧 `scopeEpoch` 时，重试必须返回原领域结果与原 `firstResult`，同时标记 `historical = true`、`snapshotRequired = true`；客户端必须从当前完整快照取得 `scopeCursors[]`，应用旧增量、把旧 `serverSeq` 用作任何当前基线或再次执行领域逻辑的次数都必须为 **0**。
- 接受、无事件拒绝和需要多个串行键的命令都必须按协议注册表选择唯一 `primaryResultScope`；`firstResult.serverSeq` 必须等于首次裁决完成后该 scope 的游标，无事件拒绝不得省略或伪造另一 scope 的序号。
- 对同一账号并发提交两个角色创建请求时，只能成功创建一个切片角色；第二个请求稳定拒绝，半成品与重复角色数必须为 **0**。
- `say` 未启用时必须以稳定的能力不可用结果拒绝且永久状态变化为 **0**；启用时的完整隔离和非持久化测试见 §6.4。

### 5.3 战斗与成长

- 相同初始状态、命令序列、时钟和随机种子重复执行 **100 次**，最终状态和结构化事件序列必须完全一致。
- 战士和法师分别使用基准装备、推荐默认战术，对每种不会逃离的首切片普通敌人模板至少运行 **30 个种子**；Young Wolf 不纳入该胜率样本，单独按下一条逃离契约验收。
- 不会逃离的同级普通战自动胜率应不低于 **90%**；固定种子样本中位时长必须为 **15–35 秒**、90 分位不超过 **60 秒**，任何一场在 **120 秒**后仍未结束都计为失败。
- 战斗结束后不得自动选择下一名敌人；自动连战次数必须为 **0**。
- Worker 呼救和 Garrick 把空闲 Thug 加入本场分别验证一次多敌人加入、确定性自动选目标、`reinforcement_joined` 决策帧和逐敌人奖励；无关空闲敌人被自动拉入的次数必须为 **0**。
- Garrick 到达 50% 生命时若 Thug 已经属于本场，该 Thug 必须复制 Garrick 在触发瞬间的合法当前目标并最多强制集火 8 秒；目标死亡、逃离或失效时立即回到正常威胁选择，结果在相同事件序列下必须一致。该分支只广播一次立即生效的目标改变提示，不设置响应窗口；`reinforcement_joined` 事件和决策帧数量必须均为 **0**。
- Young Wolf 逃离路径至少验证一次：逃离成功不产生任何经验、计数或掉落，实例移除后按刷新时间重建，作为最后一名敌人逃离时遭遇正确结束；逃离产生的重复奖励必须为 **0**。
- Young Wolf 进入 `escape_committing` 后，对它的攻击／选目标必须稳定拒绝；若它是最后敌人，玩家 participant 仍为 active 且受本场控制，`equip` 同样拒绝，快照继续读取当前 runtime projection 而不是数据库旧生命／资源，直到无奖励终态或 `escape_abort` 事务把 participant 置为 `closed`。
- 战士与法师分别以 4 级基准状态、#18 武器和推荐战术运行“先清 NS10 守卫、再在刷新前攻击 Garrick”至少 **30 个种子**，单人完成率必须不低于 **90%**；守卫仍存活时的 Garrick 战斗另按双人推荐路径验证。
- 每个 `(SpawnInstanceId, RewardEpoch)` 的经验、任务进度、任务物品和个人掉落必须作为该敌人的结算单元原子成功或原子失败；该原子边界不扩展为整场多敌遭遇，前一敌人的已提交结果不能因后一敌人结算失败而回滚或重发。
- 非法装备、背包已满、重复购买唯一装备、非法出售、重复学习、资源不足或等级不符的训练请求不得改变永久状态。
- active 战斗参与者执行 `equip` 必须返回 `COMBAT_ACTION_NOT_ALLOWED` 且状态不变；已经 `fled`／`recovered`、仅等待旧奖励的角色可以装备，随后旧奖励提交不得覆盖装备、位置或当前生命／资源。
- 最终敌人 0 HP 后暂停奖励事务，存活 participant 在 `resolving` 窗口仍为 active；该窗口内 `equip` 同样必须返回 `COMBAT_ACTION_NOT_ALLOWED`，直到最终奖励或 abort 事务原子把它置为 `closed`。
- 角色创建必须只学会本职业两个 1 级 `abilityId`；2–5 级训练逐个写入对应稳定 ID。学习 II 级伤害技能后，旧等级只保留历史可读状态、不能再排入动作；该角色个人 `TacticsLoadout` 对应行和固定 `afk_profile` 的已解析动作引用必须在同一训练事务中迁移到新 ID，同时保留个人行开关、顺序、阈值和 `tacticsRevision`，且不得修改其他角色或全局模板。
- 命令 envelope、`queued/started/resolved/cancelled/replaced` 事件、已训练状态、个人 `TacticsLoadout` 与固定 `afk_profile` 的已解析动作必须引用 04 的精确 `abilityId`，不得用本地化名称、表格序号或客户端自造别名；引用错误数必须为 **0**。
- 按 04 §10.2 的固定击杀、全部灰物领取并出售口径运行经济分布测试：总收入中位数必须为 **140–150 铜**，第 10–90 百分位必须落在约 **130–158 铜**；报告必须同时验证训练（50 铜）与基准加固手套（12 铜）后的战士结余区间为 **68–96 铜**，法师再计可选腰带（12 铜）后结余区间为 **56–84 铜**。

### 5.4 奖励唯一性

- 对任务交付、敌人击败、拾取和升级奖励分别执行重复请求、超时重试及“已提交但响应丢失”测试。
- 每类至少运行 **100 个故障注入样本**；重复经验、重复任务进度、重复物品和重复金币的总数必须为 **0**。
- 同一奖励唯一键只能存在一条有效记录；已终结的奖励代次重新加载后不得再次发放。
- 每个随机掉落的 `lootTableId` 必须精确等于对应 mob 模板记录 ID，随机键精确包含 `(content_version, SpawnInstanceId, RewardEpoch, characterId, lootTableId)`。相同键重试必须得到相同结果；分别改变其中任一字段时必须进入独立随机身份，不能误命中旧结果。
- 普通 `PersonalLoot` 只允许 `pending → claimed` 或 `pending → declined`；领取和放弃都必须校验所有者、支持相同命令幂等重试，并使结果进入不可逆终态。`declined` 后再次领取或放弃不得产生资产或第二次状态变化。
- 任务物品不得出现在 `PersonalLoot` 状态机中；对任务物品执行 `loot`／`decline-loot` 必须拒绝且任务物品、目标进度均保持不变。

### 5.5 已冻结边界用例

- 客户端帮助、补全和快捷操作不得暴露 `retreat`；提交文字 `retreat` 或 `combat.retreat` 必须返回 `ProtocolErrorResult(UNKNOWN_COMMAND_TYPE)`，不带 `firstResult`，不得被别名层映射为 `flee`，且接战关系、位置、生命／资源与动作队列变化均为 **0**；`approach` 和显式 `flee` 仍按各自规则工作。
- 让法师分别在在线 `active_profile`、断线 `afk_profile` 和最终事务暂停的 `resolving` 窗口连续 15 秒不消耗法力，期间跨过多个五秒与两秒边界：三种状态的被动法力恢复必须均为 **0**。升级产生的最大／当前法力正向差值仍按成长规则结算，不能被误判为五秒回蓝。
- 在玩家火球术或寒冰箭读条期间分别注入一次非致命直接伤害和一次持续伤害：读条不得取消，`resolvesAt` 与原读条时长必须完全不变，并在原时刻恰好完成一次。另构造伤害与施法结算同毫秒、两种 `eventSeq` 顺序的样本，仍按 04 §8 唯一顺序得出一次结果；致命伤害、0 HP 与明确打断仍按规则取消。敌人 Thug 包扎受直接伤害中断的专属行为保持不变。
- `tactics.view`／`tactics.update` 按 [05 §4](./05-tactics-and-decisions.md) 注册：view 必须返回 `contentVersion`、配置版本 `tacticsRevision`，且每行返回稳定 `ruleId`、当前 `thresholdBp` 与 `allowedThresholdBps[]`；结构化更新和文字 `<rule>` 只接受该 ID，显示行号与本地化标签不能误命中其他规则。脱战合法编辑逐次递增并持久化配置版本；战斗控制（含 `resolving`）与死亡状态提交编辑必须返回 `TACTICS_EDIT_NOT_ALLOWED` 且零变化；固定行禁用／移动、越界整数位置、非枚举阈值分别命中 05 §4.2 错误码。
- `tactics.update` 的五种 `op` 分别通过判别联合 Schema；缺字段、`position = 2.5`／`thresholdBp = 1000.5` 等非整数，或携带当前 `op` 不允许的额外字段，稳定返回 `ProtocolErrorResult(INVALID_PAYLOAD)`。以火球为原始载荷样本时，`thresholdBp = 1_000` 合法，10 必须返回 `TACTICS_INVALID_THRESHOLD`；文字 `tactics set rule_mage_fireball 10` 必须展开为 1,000 后再提交。无参数 `reset` 从编辑态恢复模板默认的开关、顺序、阈值并只递增一次版本；默认态再次 reset 返回 `changed = false`。给 `reset` 携带 `ruleId`、`position` 或 `thresholdBp` 必须在协议层拒绝，以上拒绝均零领域变化。
- 把 04 §6.2／§6.3 的战士、法师 active／afk 表做逐行参数化测试，并对 active 表额外跑一份编辑后配置（至少含禁用一行、移动一行、修改一个阈值）：逐项令更高优先级为假、当前行为真，必须选择该行精确 ability；阈值两侧、能力替换和全部候选非法时的回退均与当前生效行序一致。法师火球 active 保留量在 1–5 级及至少一件加最大法力装备的样本中必须按当前最大法力的 0%／10%／20%／30% 交叉相乘判断，不得按固定点数或先取整解释；`maxMana = 130` 时，active 10% 的 `manaAfterCost = 13` 通过而 12 不通过，固定 afk 25% 的 33 通过而 32 不通过。
- afk 表没有任何编辑入口或服务端操作，其条件、顺序、阈值和 `(class, profileKind, content_version)` 在所有编辑前后均不改变。分别参数化战士英勇打击 II 与法师火球术 II，并为每个职业创建同职业角色 A、B：A 学习后，只有 A 的个人战术行和固定 afk 已解析动作引用在同一事务切到新 ID；A 的行开关、顺序、阈值、`tacticsRevision` 以及 B 和全局模板均不变。A 随后断线接管必须实际选择 II 级动作；按“编辑 → 学习 II 级 → reset”执行后仍引用 II 级，reset 只恢复配置默认并按实际变化递增一次版本。部署测试 `content_version` 时配置重置、新版本内 revision 回到 0，但 A 的 active／afk 仍按已训练状态解析为 II 级，并只提示一次。同一配置版本下分别用 R1、R2 开始 Attempt，`CombatStartCheckpoint.activeTacticsRows[]` 与 `afkResolvedActionRefs[]` 必须按规范顺序保存并只在对应精确 `actionRef` 上不同；后续编辑、训练、奖励或 abort 均不能回写旧检查点。
- Worker 呼救和 Garrick 实际加入空闲 Thug 各构造一次：每名仍受本场控制的在线玩家恰好收到一个含稳定选项映射、5 秒倒计时和唯一默认项的 `reinforcement_joined` 决策帧；选择、超时、断线分别按 [05 §5](./05-tactics-and-decisions.md) 得出唯一结果。Garrick 只对已在场 Thug 启动强制集火时，系统把该 Thug 的强制目标设为 Garrick 在触发瞬间的合法当前目标，并广播一次立即生效的目标改变提示；不设置响应窗口，事件和决策帧均为 0。另分别构造 Worker 没有空闲 Kobold、Garrick 的 Thug 属于其他 `CombatSession`、没有存活合法 Thug；每个样本新增敌人 participant、`reinforcement_joined` 事件和决策帧数量均为 0。
- 对普通手动覆盖和战士“下一次挥击”覆盖各自构造 `queued`、`started` 两态，共 4 组：`queued` 组必须恰好一次转为 `cancelled(reason = DISCONNECT_PROFILE_SWITCH)`，资源／GCD／冷却变化为 **0**，随后由 `afk_profile` 重新选择；`started` 组在角色与目标持续合法时保留原 `resolvesAt`、成本／GCD／冷却恰好发生一次并恰好结算一次。断线转换按 04 §8 属于 priority 10；与尚未开始的选择／命令同一 `dueAt` 时必须先取消，只有更早调度键已经进入 `started` 的动作才继续。重连、重复断线通知和旧连接迟到命令都不得复活或重复动作。
- 法师在至少缺少 60 法力时使用一份魔法水：使用瞬间法力变化为 0，`+1,000ms` 到 `+10,000ms` 各恰好恢复 6，累计 60。分别在第 5 跳后成功移动、成功进入战斗，后 5 跳必须取消且不返还物品；失败移动／开战命令不得取消。令取消命令恰好与第 5 跳同毫秒到达时，第 5 跳先结算。按“战斗状态 → active timer → 满法力”的顺序验证拒绝码；另构造 timer active 且已满的重叠态，必须返回 `RECOVERY_ALREADY_ACTIVE`。所有拒绝的物品消耗数均为 0。
- 对魔法水补充故障与上限测试：相同 `commandId` 重试只消耗 1 份且只创建 10 个 tick；接近上限时逐跳截断，之后耗蓝不补旧溢出；30 秒断线宽限内 tick 继续且同进程重连可见。主动退出／宽限结束时先持久化已结算法力再取消剩余 tick，重新进入后 timer 不恢复。人为延迟库存事务提交时，`usedAt` 必须取提交成功返回队列的时刻，整组 `+1..+10` 秒相应后移且提交后补跑数为 0。成功使用后注入进程崩溃，物品消耗保持、尚未落库的法力增量和剩余 timer 不恢复；重启后重试首次命令不得重建 timer。

## 6. 集成验收

### 6.1 协议、数据库与事务

- 从空数据库执行全部迁移并装载冻结内容，成功率必须为 **100%**。
- 服务端只有在永久事务提交成功后才报告永久奖励成功；提交失败时客户端成功确认数必须为 **0**。
- 快照必须从唯一逻辑 `CharacterState` 生成基础字段，且必须包含 [01 §2](./01-player-flow.md) 的全部字段与派生 `slice_complete`。在线非战斗角色以唯一 live 实例为准，PostgreSQL durable snapshot 只作为崩溃恢复点；角色受 active 战斗控制时，仅所属 session 的唯一运行时投影可以覆盖生命／职业资源并追加瞬态字段。同进程重连读到数据库旧值、另一场投影或奖励账本旧余额的次数必须为 **0**，投影整体反写 `CharacterState` 的次数也必须为 **0**。
- 先让非战斗角色发生至少一个已冻结节奏的脱战回复 tick，并另以 §5.5 的魔法水精确节奏各执行一次，但不触发安全离线提交：同进程立即快照和断线重连必须显示 live 新值；随后注入进程崩溃必须恢复到最近 durable snapshot。另在未落库 live 回复后延迟提交旧战斗奖励，事务必须以 live 当前值应用升级并在同一提交中持久化合并结果；用数据库旧值覆盖回复的次数必须为 **0**。
- 让 `fled` 或 `recovered` 角色带未决冻结奖励进入安全离线并卸载 live 实例，再令旧奖励提交：事务必须在角色执行权内从最新 durable snapshot 唯一装载、合并、提交并可再次卸载。与重连并发运行时只允许一个顺序化结果；出现两份 live 实例、奖励丢失、旧值覆盖或重连漏结果的次数必须为 **0**。
- 模拟事务提交前崩溃、提交后广播前崩溃和仅内存战斗状态丢失三类窗口，各运行至少 **20 次**，恢复结果必须符合冻结的恢复契约。
- 角色在目标死亡前 `recover` 并处于 `COMBAT_REWARD_PENDING` 时注入进程崩溃：重启后角色保持庭院满资源状态、等待标记清除、未提交奖励不补造，重复奖励数为 **0**。
- 对已 `recover` 的等待者延迟提交旧战斗奖励时，奖励只能在当前 `CharacterState` 上追加经验、任务进度、任务物品、铜币和 `PersonalLoot`；庭院位置、满生命／资源及恢复后的装备变化被旧参与者快照覆盖的次数必须为 **0**。
- 非最终敌人的奖励使仍在战斗的角色升级时，事务后的 active combat runtime projection 必须在下一事件前同步新等级、最大值和当前值增量；继续使用旧上限或反向覆盖 `CharacterState` 的次数必须为 **0**。
- 最终敌人的奖励恰好使角色升级时，仍为 active 的 recipient 必须以“奖励前一刻运行时值 + 本次升级正向差值”一次写入生命／法力并受新上限约束，怒气保持运行时值；同一双人样本中的 `defeated`／`recovered` recipient 必须在当前 `CharacterState` 上应用升级，死亡者生命保持 0、法力仍增加本次 `levelUpManaDelta` 并封顶，恢复者庭院和满资源不被旧 runtime 覆盖。任何路径被旧投影覆盖的次数必须为 **0**。
- 在上述升级已经提交后模拟进程崩溃或后续敌人 `reward_abort`：active 角色必须按开始检查点加已提交最大值／当前值增量重基准恢复，等级保留但 current-value 增量丢失的次数必须为 **0**。
- 在敌人 0 HP 后、奖励提交前分别执行“才接取匹配任务”和“放弃后重接为新 `questRunId`”：两者都不得从旧击败获得任务计数或任务物品；战斗经验和普通 PersonalLoot 仍按冻结战斗资格结算。
- 角色以 `defeated` 状态崩溃或重启后仍须保持死亡，直到玩家成功执行 `recover`；自动复活、自动移动或丢失死亡原因的次数必须为 **0**。
- 分别在“敌人奖励已提交但刷新尚未到时”和“Young Wolf 已逃离但刷新尚未到时”重启服务端；原 `SpawnInstanceId` 不得重新开放，持久 `respawnEligibleAt` 到达前提前刷新的次数必须为 **0**，刷新后必须使用新的 `SpawnInstanceId` 和 `RewardEpoch`。
- 对普通奖励事务注入连续 3 次失败：`reward_abort` 事务必须原子完成 active 角色重基准写回、participant `closed`、attempt 关闭、未提交冻结结果丢弃和 `unresolvedRewardEpochs`／reward lock 清理；提交前仍属旧 session 的实例不得被第三方认领，提交后较早已提交敌人结果保留，仍属旧 session 的非终态敌人以原实例／活动 RewardEpoch 重置并且以后只可结算一次。
- 对 Young Wolf 无奖励终态事务注入连续 3 次失败：不得广播逃离或生成刷新；`escape_abort` 使用相同原子关闭边界，只重置仍属旧 session 的非终态实例，随后正常击败只结算一次，中止事务提交前不得解除 session 锁定。
- 对两类 abort 都先建立加入前 Battle Shout／Frost Armor 与 ability 专属冷却，再在 attempt 内分别新建、刷新并自然到期：捕获必须发生在 participant 创建后、首个动作成本／效果前，并排除 GCD、挥击／读条／移动计时和目标 debuff。恢复后只保留当前时刻仍有效的加入前效果及其原到期时间，本 attempt 新建效果残留、过期效果复活或持续时间倒退的次数必须为 **0**。
- 两类 abort 的终局摘要在中止事务提交前必须为 **0 次**、提交后恰好 **1 次**；模拟提交后响应丢失再重试不得重复广播，`escape_abort` 摘要不得显示敌人成功逃离。
- 未完成战斗进程重启后，非终态敌人必须保留原来仍为 active 的 `SpawnInstanceId/RewardEpoch`、旧 EnemyParticipant 作废，且只有下一次合法攻击才创建新 session/attempt；随后正常击败的总奖励次数必须为 **1**。

### 6.2 多人一致性

- 3、4、5 个客户端分别执行第 8 节组合；所有参与客户端最终可见状态必须与服务端权威快照一致。
- 同一 `(scopeEpoch, scope)` 内的事件序号不得倒退或重复应用；发现缺口后必须通过完整同步恢复。
- 服务端重启或作用域代次变化时，完整快照必须携带每个可见 scope 的新 `scopeCursors[]` 项，客户端逐 scope 清空旧序号基线后再应用新代次事件；新代次合法事件因与旧 `serverSeq` 数值相同而被误去重的次数必须为 **0**。
- 同一快照同时携带 character scope 与 instance scope，令二者使用不同 epoch／序号并分别制造缺口；任一 scope 的游标被套用到另一 scope、或一个缺口错误暂停另一个 scope 的次数必须为 **0**。
- 在角色跨房间移动与重连快照并发时，服务端必须按稳定 scope 顺序取得短读屏障，并在同一数据库一致性快照中捕获状态与 `scopeCursors[]`；角色位置／房间成员状态和 cursor 不属于同一切面的次数必须为 **0**。数组必须按 scope 排序且无重复，已离开、当前不再可见的旧 scope 必须从完整快照和客户端基线中删除。
- 在快照切面捕获后、客户端收到快照前注入同 scope 新事件：新连接的订阅绑定与快照入 FIFO 发送队列必须发生在释放读屏障前，增量只能排在快照之后且恰好送达一次。增量先于快照、交接空窗丢失、旧连接事件进入新队列或失败交接残留半订阅的次数必须为 **0**。
- 同一房间内的进入、离开、移动和相关战斗事件，在所有应当可见的客户端上内容一致；最终状态分歧数必须为 **0**。
- 不同角色的任务、知识、背包和成长状态不得互相覆盖；串写事件数必须为 **0**。
- 共享敌人、任务计数、经验与掉落直接按 [06 §2.2](./06-multiplayer-and-recovery.md)／[§3.1–§3.3](./06-multiplayer-and-recovery.md) 已冻结的归属规则验证，不能只验证“没有报错”。
- 同一 session 累计登记 A、B 后，无论其中一人已 `defeated`、`recover` 或 `flee`，第三名 C 攻击都必须收到 `COMBAT_PLAYER_CAPACITY_REACHED`；已逃离成员重新加入同一 attempt 的成功次数必须为 **0**。
- 构造旧 session 已失败但仍有奖励等待的三客户端场景：存活敌人 `reset_detached` 后由 C 的新 session 认领，再让旧奖励进入 `reward_abort`；旧中止改变 C 的归属、生命、临时状态或事件序列的次数必须为 **0**。

### 6.3 断线与重连

- 非战斗状态下，在移动前后、任务推进前后、奖励提交前后和装备变化后分别断线重连。
- 战斗状态下，在普通动作、覆盖动作待执行、关键意图已公布和奖励提交前后分别断线重连。
- 每个位置至少运行 **10 次**；数据丢失、状态回滚、命令重复执行和重复奖励必须全部为 **0**。
- 新连接恢复控制后，旧连接继续修改角色状态的成功次数必须为 **0**。
- 旧连接在接管前已入队但尚未开始执行的命令必须携带旧 `controlEpoch`，并在真正执行前再次校验；新连接接管后，这些旧队列命令成功改变角色状态的次数必须为 **0**。
- 断线后由 `afk_profile` 新选择的服务端自动动作不携带客户端 `controlEpoch`，但必须在开始前验证当前参与者／profile；它因旧连接 epoch 被误取消或冒充旧连接命令的次数必须为 **0**。
- 重连后的当前状态和摘要必须来自服务端权威快照；客户端本地旧状态不得覆盖服务端结果。
- Battle Shout／Frost Armor 及技能冷却在同一进程的断线重连中按内存剩余时间保留并出现在快照；`recover` 只清除战斗专属 aura、仇恨和目标，不清除仍有效的上述自我增益；安全离线后重新进世界或进程重启则必须清除这些非持久状态。

断线瞬间尚未开始的手动覆盖必须按 §5.5 取消为 `DISCONNECT_PROFILE_SWITCH`；已经开始的动作不因断线取消，并在角色／目标持续合法时保持原 `resolvesAt`、恰好结算一次。旧 `controlEpoch` 命令仍不得在新连接接管后越权执行。三条边界必须在同一故障注入组合中同时成立，不能用接管取消替代断线 profile 切换取消。

### 6.4 可选 `say` 能力

- 能力关闭构建只验证稳定拒绝、状态不变及不影响 `sliceCompleteAt`；不因缺少 `say` 判定切片失败。
- 能力开启构建必须验证同房间实时可见、其他房间不可见、断线期间不补发，且聊天正文写入数据库、完整快照、未读摘要或普通验收日志的条数均为 **0**。
- 发送者离开房间、断线或重连后，不得从服务端历史恢复先前聊天；聊天消息也不得改变任务知识或前置。

## 7. E2E 验收

### 7.1 单人黄金路径

用真实客户端执行一次不调用管理员能力的完整路径：

1. 进入或恢复角色；
2. 查看房间和出口；
3. 完成 #783，并接取 #7 与 #18；
4. 依次完成 #7、#15、#21，并接取 #54；
5. 完成 #18、#6、#3903、#3904、#3905；#6 按“先清守卫、再在刷新前挑战 Garrick”的单人基准路径执行；
6. 在任务要求的战斗中至少完成一次个人掉落领取；获得该职业全部任务装备，并穿戴 04 第 10.3 节的最终基准组合；
7. 无额外刷怪达到 5 级，购买并装备 `item_reinforced_gloves`，完成当前等级全部付费训练；
8. 退出并重连，确认任务、装备、训练与位置恢复；
9. 到达南路的闪金镇方向边界并首次写入 `sliceCompleteAt`；
10. 向北返回后再次退出重连，确认 `sliceCompleteAt` 不变、派生 `slice_complete` 仍为 true 且当前位置准确恢复。

要求：流程中无软锁、无管理员修复、无丢档、无重复奖励，且所有永久结果能在重连后恢复。

### 7.2 非法路径

至少覆盖：

- 跳过任务前置；
- 重复交付或重复领取；
- 背包或目标状态不允许时继续操作；
- 对已死亡、不可见或不合法目标提交动作；
- 在命令响应丢失后重复提交；
- 使用旧连接继续控制已被新连接接管的角色。

每个非法路径都必须返回可识别结果，永久状态变化次数必须为 **0**。

### 7.3 死亡恢复与奖励资格边界

候选发布构建必须用真实客户端跑完七条独立路径：

1. 单人战斗在目标被击败前到达 0 HP，确认普通移动、任务和攻击被拒绝；旧 `Attempt` 关闭且未冻结资格被清除。执行 `recover` 后满生命／职业资源返回修道院庭院，重复提交不产生第二次变化。
2. A、B 合法加入同一战斗，A 到达 0 HP 后在目标被击败前执行 `recover`；A 此时开启新战斗应收到 `COMBAT_REWARD_PENDING`。B 随后击败目标；A 对死亡前已登记的该目标仍保留资格，A、B 各自只获得一次经验、任务计数和个人掉落，结算后 A 可再次正常开战。
3. A、B 合法加入同一战斗，先让 A 受伤并消耗法力／怒气，再成功 `flee`；目的房间、完成瞬间生命／资源和 participant `fled` 必须原子提交，提交后立即快照及同进程重连不得回到战前数据库值。随后 B 击败目标；A 对该目标获得的经验、任务计数、铜币和物品必须全部为 0，B 正常结算一次。另注入逃离事务失败，A 必须留在原房间并保持 active 投影，不得出现半移动。
4. A、B 在任何敌人到达 0 HP 前均到达 0 HP 或全部成功退出；旧 `CombatSession`／`Attempt` 直接关闭并清除未冻结资格，敌人重置后由其他角色击败时，A、B 获得的旧资格奖励必须为 0。
5. 敌人降至 0 HP 并冻结 `eligibleRecipients` 后、奖励提交前，让合格角色执行 `recover` 或随后断线；结算仍严格按冻结快照各一次，既不移除合格者，也不新增未登记角色。
6. 分别为战士／法师施加 Battle Shout／Frost Armor：死亡后 `recover` 与同进程断线重连必须保留剩余持续时间，而战斗专属 aura、仇恨和目标必须清除；完成安全离线再进入或重启进程后，自我 aura 和 ability 专属冷却必须清除。
7. A、B 在多敌人战斗中先使一名敌人到达 0 HP，并暂停其奖励提交；A 随后 `flee` 到相邻房间，尝试开启或加入另一场战斗必须收到 `COMBAT_REWARD_PENDING`。旧冻结奖励成功或中止事务提交、且 A 没有其他保留资格后，新战斗才允许开始；旧奖励不得覆盖新位置或其他当前状态。

七条路径都必须验证退出重连后的权威快照；任何自动复活、资格误判、跨 session 奖励锁提前释放、错误 aura 生命周期或重复奖励均为失败。

### 7.4 `NS-SLICE-01` 切片完成专项

候选发布构建必须用真实客户端验证：

1. 尚无 `sliceCompleteAt` 且 #54 非 `active` 时，从 NS11 执行 `go south` 不写入标记；
2. #54 为 `active` 时首次执行只写入一次 `sliceCompleteAt`，#54 保持 `active`，角色仍在 NS11，任务交付、XP、铜币和物品奖励变化均为 **0**；
3. 使用相同 `commandId` 重试及使用新 `commandId` 再次触发，都返回原完成摘要且首次时间不变；
4. 完成后放弃 #54 会清除 Documents 但不清除 `sliceCompleteAt`；重接只重建一份 Documents，再次触发仍返回原完成结果；
5. 退出重连后快照携带原 `sliceCompleteAt`，客户端派生 `slice_complete = true`，位置和 #54 当前状态均准确恢复。

任一步产生本地 #54 交付、第二个完成时间、额外资产或跨入闪金镇房间，都判定 `NS-SLICE-01` 失败。

### 7.5 六项冻结战斗边界专项

候选发布构建必须通过真实 Web 客户端、实际协议连接与旁观客户端完成：

1. 帮助、补全和快捷操作均不出现 `retreat` 或 afk 表编辑；实际输入 `retreat`、`combat.retreat` 均稳定拒绝，且 `retreat` 绝不触发 `flee`。真实客户端用 `tactics.view` 返回的稳定 `ruleId` 完成一次合法编辑并在重连后保持；显示行号和本地化标签均不能充当身份。随后执行无参数 reset 恢复配置默认；缺字段、非整数位置、错误额外字段和带参数 reset 均显示协议拒绝且零变化，战斗中编辑同样稳定拒绝。
2. 法师分别在在线、断线接管和最终事务暂停窗口跨过 15 秒，客户端与服务端事件中都没有被动正向法力变化；在火球术／寒冰箭读条中承受非致命伤害后，旁观端看到法术在原 `resolvesAt` 恰好完成一次。
3. 分别在普通覆盖和“下一次挥击”覆盖仍为 `queued`、已经 `started` 时断线；旁观端与重连快照共同证明前者无成本取消并由 `afk_profile` 接管，后者在合法样本中保持原时刻且恰好结算一次。
4. 魔法水完整 10 跳与第 5 跳后成功移动取消各跑一次，逐跳事件、背包数量、同进程断线重连快照和最终法力均符合 §5.5；失败移动不得取消，文字输出不得把 10 秒恢复误报为瞬时恢复。
5. 战士与法师分别完成 II 级替换训练；个人战术与断线接管实际使用新等级，同职业未训练角色仍使用原等级。训练后 reset 不回退 ability ID，也不改写全局模板或其他角色。
6. Worker 呼救与 Garrick 加入空闲 Thug 各通过真实客户端产生一次 5 秒决策帧；旁观端证明每名受控在线玩家每次实际加入恰好一帧。再验证 Garrick 已在场 Thug 的分支只把该 Thug 的强制目标设为 Garrick 在触发瞬间的合法当前目标，并广播一次立即生效的目标改变提示；没有响应窗口、`reinforcement_joined` 事件或决策菜单。

任一路径被客户端别名、预测或本地缓存掩盖服务端结果，都判定本专项失败。

### 7.6 完整切片冒烟

每个候选发布构建至少跑一次从进入北郡到切片出口的 E2E 冒烟。它可以使用固定脚本加速，但不能替代真人 60–90 分钟试玩。冒烟失败时不得进入正式真人试玩。

## 8. 3–5 人多人测试组合

以下组合都使用独立客户端和独立角色。共享奖励结果只验证 [06 §2.2](./06-multiplayer-and-recovery.md)／[§3.1–§3.3](./06-multiplayer-and-recovery.md) 已冻结的规则，不在本文发明归属策略。

### 8.1 三人最小组合

- A、B 进入同一房间并参与同一场战斗；
- C 在同一世界实例中独立移动、查看环境或推进自己的任务；
- 战斗期间让 B 断线并重连；
- 验证 A、B 的战斗结果一致，C 的私有进度不被覆盖，三人的房间成员状态最终收敛。

### 8.2 四人交叉组合

- A、B 参与一场战斗；C、D 在另一房间参与第二场独立 `CombatSession`，并在两场战斗前后各执行一次移动或任务操作；
- 让两组在测试中途交汇并再次分开；
- 对一个有状态命令模拟响应丢失和相同幂等 ID 重试；
- 验证两场活动互不污染、交汇时可见状态正确、重复永久结果为零。

### 8.3 五人高交互组合

- A、B 参与同一场战斗；C 旁观该战斗但攻击时应收到双人容量拒绝；D、E 在相邻流程中移动或与 NPC 交互；
- 同时触发一次战斗覆盖动作、一次房间移动和一次断线重连；
- 至少一名战斗参与者在奖励广播前后重连；
- 验证权威事件顺序唯一、所有客户端最终快照一致、角色私有状态隔离、重复奖励为零。

每种组合至少连续运行 **10 轮**。任何状态分歧、串档、重复奖励或旧连接越权都属于 P0 失败。

## 9. 真人试玩方案

### 9.1 参与者

每轮使用 **3–5 名**未玩过当前构建、未阅读本切片路线答案的测试者。记录但不强行统一其 Vanilla 和 MUD 经验；结果按“熟悉 Vanilla”“不了解路线”“熟悉文字命令”分组查看，避免平均数掩盖新手问题。

正式完成门槛优先使用 5 人样本；只有 3–4 人时可以进行方向性试玩，但报告必须标明样本不足，后续补足到累计至少 5 个独立完整流程。

### 9.2 测试方式

- 测试者从相同规则的标准初始状态开始，但使用独立角色。
- 第一轮先独立推进；多人交互只在自然相遇或测试脚本指定的短场景中发生。
- 主持人只观察和记录，不解释任务答案或路线。
- 测试者可以使用游戏内已提供的 `look`、出口、任务日志和帮助能力。
- 每次口头求助都记录时间、当前位置、最近目标、已经尝试的命令和最终提示。
- 完成后进行不超过 15 分钟的访谈：最清楚的目标、最迷惑的节点、战斗中主动接管的理由、成长是否可感知、多人状态是否可信。

## 10. 真人试玩量化门槛

### 10.1 完成时间

- 从角色可操作到到达切片出口，目标为 **60–90 分钟**。
- 一轮 3–5 人测试中，完成时间中位数必须落在 **60–90 分钟**。
- 至少 `ceil(参与人数 × 80%)` 的测试者应落在该区间；3 人样本即至少 3 人，4–5 人样本即至少 4 人。
- 成功完成者不得超过 **105 分钟**；因软件故障导致的等待单独记录，不得从结果中静默删除。

### 10.2 任务完成率

- #783、#7、#15、#21 的交付以及 #54 的接取完成率必须为 **100%**；任何无法继续的主任务都属于软锁。
- 已接取任务的完成率应不低于 **90%**。
- 任务完成率按“成功交付数 ÷ 已接取且在切片内可完成的任务数”计算。
- 管理员直接推进、补发目标或修改状态后完成的任务不计为正常完成。

### 10.3 迷路与求助

“迷路事件”定义为：玩家连续 **3 分钟**没有朝当前目标产生位置或任务进展，并明确表示不知道去哪里，或在同一小段路径无目的往返两次以上。

- 每名测试者的迷路事件中位数应不高于 **1 次**。
- 单次迷路不得持续超过 **5 分钟**。
- 至少 **80%** 的测试者应在没有主持人路线提示的情况下到达切片出口。
- 主持人路线提示总数应不高于 **1 次/人**；任何为解除软锁而进行的管理员干预都直接判为失败。
- 内置帮助使用单独计数，不等同于失败；若同一节点有至少 2 名测试者请求相同帮助，则必须进入设计复盘。

### 10.4 无效命令

“无效命令”包括无法解析、缺少必要参数、目标不存在或当前状态不允许执行的命令。玩家主动尝试合理但未实现的同义词也计入，用于发现交互缺口。

- 前 10 分钟无效命令率应不高于 **15%**。
- 第 10 分钟后无效命令率应不高于 **5%**。
- 至少 **95%** 的无效命令应返回可操作反馈，说明错误类型或下一步合法形式。
- 同一测试者连续 3 次因同一原因失败且系统未提供更清楚提示，记为一个交互阻塞点。
- 所有无效命令造成的未声明永久状态变化数必须为 **0**。

### 10.5 战斗胜率与时长

对使用基准装备和推荐默认战术、不会逃离的同级普通战（不含 Young Wolf）：

- 真人试玩总胜率应不低于 **90%**。
- 同级普通战的中位时长应为 **15–35 秒**；超出时即使未到硬超时也进入节奏复盘。
- 单场普通战不得超过 **120 秒**；超过即记为战斗失败或停滞。
- 战斗结束后自动进入下一场的次数必须为 **0**。
- 死亡必须能从摘要中追溯到明确原因；“不知道为什么死”若被至少 2 名测试者独立报告，则体验门槛失败。

Young Wolf 只按 §5.3 的逃离结果、遭遇终止、零奖励和刷新恢复单独统计，不以逃离结果降低普通敌人胜率。

### 10.6 主动接管收益

使用相同初始状态和成对随机种子，对“只使用默认战术”和“允许玩家主动覆盖”各运行至少 **10 对**普通战：

- 主动接管后的胜率不能低于默认战术；并且
- 战斗时长、结束生命或关键资源消耗中，至少一个指标的中位改善达到 **15%**；并且
- 测试者能说出自己为何接管以及该动作造成了什么结果。

若默认战术已经稳定获胜但主动接管没有可观察收益，判定战斗决策价值不足；不得通过故意削弱默认战术到频繁死亡来制造收益。

同一轮同时记录战术入口与关键决策帧的可理解性：首次查看 `tactics` 后，至少 **80%** 测试者能指出默认自动动作、说明查看与编辑都不是开战前置并知道 reset 只恢复配置默认；首次看到 `reinforcement_joined` 帧后，至少 **80%** 能在不求助下指出默认项，并理解 5 秒超时会继续原战术而不会停战。未主动选择或错过窗口不得破坏低注意力自动战斗闭环。

### 10.7 多人与恢复

- 应当看见同一权威事件的客户端，最终结果一致率必须为 **100%**。
- 私有任务、背包、知识与成长串写次数必须为 **0**。
- 断线重连后的永久数据丢失、回滚和重复执行次数必须为 **0**。
- 重复经验、任务进度、物品或金币总数必须为 **0**。
- 若重连期间战斗继续，恢复后的摘要必须让测试者理解期间结果；至少 **80%** 的参与者应能正确回答胜负、当前生命和是否获得奖励。

## 11. 通过门槛

### 11.1 P0 硬门槛

以下任一项失败，构建不得宣称北郡 V1 完成，也不得用体验调优掩盖：

- 主任务软锁或无法到达切片出口；
- `sliceCompleteAt` 无法首次写入、重复触发改变首次完成时间、派生 `slice_complete` 错误，或完成后重连丢失；
- 0 HP 后无法通过幂等 `recover` 返回有效状态，或出现自动复活；
- 战斗中断线后未切换到冻结的 `afk_profile`、当前遭遇未继续，或重连无法恢复权威结果；
- 权威状态与客户端最终状态分歧；
- 不同角色之间串档；
- 永久数据丢失或未声明回滚；
- 延迟奖励用旧战斗投影覆盖已恢复的角色位置、生命、资源或装备；
- 重复经验、任务进度、物品或金币不为 0；
- 无效命令改变永久状态；
- 旧连接或携带旧 `controlEpoch` 的已排队命令在重连接管后仍能控制角色；
- §5.5 或 §7.5 任一六项冻结边界专项失败；
- 任一 `scopeEpoch` 变化后客户端沿用该 scope 的旧序号基线，或把不同 scope 的游标混用，造成合法事件丢失或旧代次事件混入；
- 依赖规则仍为 `BLOCKED`，却把相关场景标记为通过。

### 11.2 体验门槛

以下项目允许失败后迭代，但必须有数据、原因和复测计划：

- 60–90 分钟完成目标；
- 已接取任务完成率；
- 迷路、求助和无效命令率；
- 普通战胜率与时长；
- 主动接管收益；
- 战斗摘要和成长反馈的理解度。

体验门槛未通过时，技术正确性可以记为通过，但整个切片仍不能记为完成。

## 12. 失败后的设计动作

| 失败信号 | 首先检查 | 必须采取的动作 | 不应直接采取 |
|---|---|---|---|
| 中位完成时间超过 90 分钟 | 导航、往返、战斗和阅读各自耗时 | 找出占比最高的两段，缩短无决策等待或澄清目标后复测 | 未定位原因就整体提高移动或经验速度 |
| 多数人在 60 分钟内完成 | 是否跳过任务、成长或理解步骤 | 检查内容是否被绕过；若闭环完整则记录为节奏候选，不自动加填充内容 | 为凑时长增加重复击杀 |
| 主任务或已接取任务完成率不足 | 前置、目标文本、刷新、任务物品和交付条件 | 修复具体阻塞点，并为该路径新增自动化回归 | 只让主持人补发进度 |
| 迷路或求助超阈值 | 房间地标、出口命名、任务方位和 `look` 信息 | 优先修改地标、相邻关系表达或任务提示；用同背景新测试者复测 | 立即加入全程箭头或直接给答案 |
| 无效命令率过高 | 缺失别名、参数顺序、错误反馈和状态提示 | 增加合理同义词、示例或情境化错误；为高频错误新增测试 | 让客户端绕过服务端校验 |
| 自动战胜率低或时长过长 | 基准装备、默认战术、资源循环和敌人数值 | 调整最小数值或默认战术，并用相同种子复测 | 先增加玩家输入频率 |
| 自动战过强且接管收益不足 | 决策点、资源保存、目标切换和覆盖结果 | 增强可理解的效率或资源取舍，保持普通战可暂离 | 故意制造随机秒杀 |
| 多人状态分歧或串档 | 状态所有权、序列化键、事件序号和快照 | 停止发布，补充最小复现与并发回归，修复后重跑全部相关组合 | 以客户端刷新掩盖服务端分歧 |
| 断线后回滚或丢档 | 提交边界、检查点和快照来源 | 修复权威恢复路径，重跑所有断线位置和故障窗口 | 信任客户端本地状态补写 |
| 出现重复奖励 | 幂等键、唯一约束、奖励代次和提交后广播 | 作为 P0 缺陷修复；增加对应故障窗口回归并确认重复数为 0 | 人工删掉重复资产后继续验收 |
| 两人以上报告相同理解问题 | 摘要、任务文本或成长反馈 | 建立一个可观察的文案或信息层修改，并安排新样本复测 | 只在试玩后口头解释 |

一次修改只解决已定位的问题。复测必须使用未接触修改说明的测试者；同一批熟悉路线的测试者只能用于回归，不用于证明首次可理解性。
