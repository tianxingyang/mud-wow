# 北郡 v1：垂直切片契约

> 状态：当前执行基线
> 适用范围：北郡 1–5 级可玩切片
> 更新日期：2026-07-16
> 关联文档：[玩家流程](./01-player-flow.md)、[产品范围](../product-scope.md)、[多人一致性](../multiplayer-consistency.md)、[战斗系统](../combat-system.md)、[系统架构](../architecture.md)

## 1. 契约目的

本文把北郡首切片的产品决定收敛为一个已冻结、可实现、可验收的范围。它回答四个问题：

1. 这次切片必须证明什么；
2. 哪些能力必须进入，哪些明确不进入；
3. 多人、掉落、死亡、断线和出区采用什么最小语义；
4. 达到什么条件才可以把切片标记为完成。

本切片不是完整游戏发行版。代码模块存在、单个接口可调用或内容已经录入，都不等同于玩家闭环已经成立。

## 2. 成功目标

北郡 v1 必须通过一次从首次进入到触发 `slice.complete`、写入 `sliceCompleteAt` 的真实游玩，验证：

- 玩家不查外部地图或攻略，也能依靠文字地标完成北郡路线；
- 任务、战斗、个人掉落、升级、训练和持久化构成最小成长闭环；
- 战士与法师在资源、读条或行动节奏上可辨识；
- 两名真实玩家可以进入同一个 `CombatSession`，并看到一致的权威结果；
- 网络断线时，角色由 `afk_profile` 完成当前遭遇，重连后恢复当前权威状态；
- 死亡不会把测试者困在未实现的跑尸流程中；
- 到达闪金镇方向的边界能够可靠提交切片完成状态，同时允许继续返回北郡游玩。

目标流程时长为约 60–90 分钟。该时长是调优目标，不是通过伪造等级、跳过任务或自动完成内容来满足的硬门槛。

## 3. 决策优先级

### 3.1 文档冲突时的优先级

北郡 v1 范围内出现冲突时，按以下顺序处理：

1. 本文负责切片范围、进入／退出边界和明确冻结的产品语义；
2. [玩家流程](./01-player-flow.md)负责玩家可见步骤、反馈、状态变化和失败恢复；
3. [世界与内容](./02-world-and-content.md)、[任务状态](./03-quest-state-pack.md)、[战斗与成长](./04-combat-and-progression.md)、[战术与关键决策](./05-tactics-and-decisions.md)及[多人与恢复](./06-multiplayer-and-recovery.md)分别负责本切片的具体内容和规则参数；
4. [验收与试玩](./07-acceptance-and-playtest.md)负责把上述规则映射为通过门槛，不得反向发明新玩法；
5. [战斗系统](../combat-system.md)、[多人一致性](../multiplayer-consistency.md)、[产品范围](../product-scope.md)、[系统架构](../architecture.md)与 ADR 负责更广的产品和工程边界；
6. 研究文档负责原版事实、任务关系和改编来源，不自动成为当前产品承诺。

00–06 共同构成 v1 行为的完整权威包，07 只负责验收。通用产品、设计和工程文档中的规则只有被对应 v1 所有者显式导入时才继承；本包未重述不等于默认采用通用规则。

不得静默选择互相矛盾的语义。若更高优先级文档没有答案，应记录为待决定事项，而不是让客户端、根层草案或实现细节自行决定游戏规则。

### 3.2 交付取舍时的优先级

在时间或复杂度冲突时，依次保护：

1. 服务端权威、奖励不可重复和角色状态不互相覆盖；
2. 玩家能完成、失败、死亡、断线并恢复的完整闭环；
3. 文字地标、任务前置和北郡事实正确；
4. 战士／法师差异与两人共同战斗；
5. 输出表现、便利功能和非必要社交。

较低优先级能力不得以破坏较高优先级语义为代价进入切片。

## 4. 已冻结的产品决定

| 主题 | 北郡 v1 决定 |
|---|---|
| 内容边界 | 只做北郡修道院／北郡山谷 1–5 级内容，闪金镇不作为可进入地图。 |
| 玩家表面 | 提供最小 Web 文字终端；只冻结输入、输出和状态恢复能力，不锁定 React、Vue 或其他前端框架。 |
| 账号 | 正常连接必须绑定权威账号／会话；每个账号在北郡 v1 只能拥有一个切片角色。测试环境可以预置账号，不要求在切片内实现公开注册、找回密码或账号管理。 |
| 首次角色创建 | 只收集角色名称和职业；职业仅可选择战士或法师。外观、背景、种族、天赋和其他创建项不进入。 |
| 多人战斗 | 两名玩家可在同一房间中对同一敌人采取合法战斗动作，从而进入同一个 `CombatSession`；不要求先建立正式队伍。每个 `CombatSession` 累计最多登记两名玩家参与者，第三名玩家可旁观但不能加入，名额计法与拒绝语义见 [06 §2.1](./06-multiplayer-and-recovery.md)。 |
| 掉落 | 使用个人掉落。每名符合资格的参与者拥有独立奖励结果，不能看见、争抢或领取另一名玩家的掉落；不做需求／贪婪。 |
| 死亡 | 使用简化恢复：死亡后可执行 `recover` 回到北郡修道院庭院，恢复满生命和职业资源，清除本场专属战斗状态；仍有效的职业自我增益按 04 保留。不跑尸、不掉经验或装备、不做耐久与复活虚弱。 |
| 断线 | 断线角色若在战斗中，立即取消尚未开始的手动覆盖、保留已经开始的动作并改用固定 `afk_profile` 继续当前 `CombatSession`；当前遭遇结束后停止，不自动移动或开启下一场战斗。 |
| 出区 | 满足切片完成条件后，在南路尝试穿过闪金镇方向边界会以幂等事务首次写入 `sliceCompleteAt`；角色位置仍为南路，且可以向北返回。 |
| 聊天 | `say` 是可选的最小能力。若实现，只在当前房间实时广播，不持久化、不补发、不进入重连摘要；缺少 `say` 不阻塞切片完成。 |
| AI | 生成式 AI 表现层在北郡 v1 关闭。所有任务、房间、战斗与错误反馈来自固定模板和结构化数据。 |

### 4.1 个人掉落的最小语义

- 敌人被击败时，服务端从该 `CombatSession` 的权威参与者集合计算 `eligibleRecipients`；首切片不增加伤害占比或反挂机门槛。
- 最小资格为：角色已经加入该 `CombatSession`，该敌人已登记到角色的资格集合，在目标被击败前没有成功 `flee`，并且该角色的奖励代次尚未提交。
- 每名符合战斗资格的角色获得完整的独立战斗经验和独立随机结果；只有 0 HP 快照已冻结匹配目标、且提交时仍是同一活动 `questRunId` 时才推进该角色的任务，随机结果也可以不生成物品。
- 一名玩家领取或放弃自己的掉落，不改变另一名玩家的结果。
- 背包已满时只拒绝该角色的领取，个人掉落记录保留为待领取；不得把物品转给另一名参与者，也不得重复生成。
- Bandana、Garrick's Head 等击杀来源任务物品在对应敌人结算事务中直接写入该角色的任务物品栏并同步推进目标；它们不是可领取／放弃的个人掉落，不受普通背包容量影响。
- 敌人降至 0 HP 时冻结其 `eligibleRecipients`；冻结后的快照不因角色随后 `recover` 或 `flee` 改变。
- 全灭或全部退出后不得重开旧遭遇，未决奖励结清后关闭；生命周期状态与转换见 [06 §2.2](./06-multiplayer-and-recovery.md)／[§7.1](./06-multiplayer-and-recovery.md)。

### 4.2 简化死亡的最小语义

- 生命降至零后，角色停止提交普通游戏命令，取消未完成读条、覆盖动作、接近和逃离过程，并退出当前战斗控制。
- 若另一名玩家仍存活，`CombatSession` 可以继续；若所有玩家均死亡，遭遇按内容定义重置。
- `recover` 是死亡状态下的唯一必要恢复动作。提交成功后，角色移动到北郡修道院庭院，恢复满生命与职业资源，并清除只属于上一场战斗的 aura、仇恨和临时目标；仍在自身持续时间内的 Battle Shout、Frost Armor 等持续型职业 aura 按 04 保留。
- 角色已持久提交的任务、经验、背包、装备和个人掉落保持不变。
- 重复 `recover` 不得重复发放收益或制造第二次状态变化。

### 4.3 `slice.complete` 的最小语义

- 首次完成的最小前置为 `Report to Goldshire` 处于 `active`，且角色位于北郡南路的闪金镇方向边界。
- 玩家尝试沿边界继续向南时，服务端在角色级幂等事务中写入唯一持久字段 `sliceCompleteAt`；`slice_complete` 只是在快照和界面中由 `sliceCompleteAt != null` 派生的布尔值，不单独持久化。
- `sliceCompleteAt` 是切片验收标记，不伪造闪金镇内的任务交付，也不额外发放经验或物品；#54 保持 `active`。
- 提交成功后不跨区移动；角色仍位于南路，并看到可向北返回的明确反馈。
- 已有 `sliceCompleteAt` 时必须先于当前 #54 状态检查短路并返回首次完成结果：之后即使放弃或重接 #54，再次尝试边界也不更新时间、不重复奖励。

## 5. In scope

### 5.1 可玩内容

- 北郡 1–5 级，按[世界与内容包](./02-world-and-content.md)冻结为 11 个互相连通、地标可辨识的房间；
- 按[任务状态包](./03-quest-state-pack.md)冻结为 10 条经核对的 Vanilla 北郡任务改编，其中 9 条可在切片内交付；
- 主干覆盖 `A Threat Within` → `Kobold Camp Cleanup` → `Investigate Echo Ridge` → `Skirmish at Echo Ridge` → `Report to Goldshire`；
- 5 类普通敌人和 1 个命名敌人；
- 战士、法师在完成对应训练后，各自在 5 级时有 5 个当前可用动作；替换前后的技能等级使用独立发布记录；
- 任务、战斗、经验、升级、装备和训练的最小闭环。

### 5.2 最小 Web 文字终端

客户端必须提供：

- 可滚动的服务端输出区域；
- 单一文字输入入口或等价的命令提交控件；
- 连接、断线、重连和命令接受／拒绝状态；
- 房间快照、出口、可见 NPC／敌人／玩家；
- 任务、战斗决策摘要、背包、装备和角色状态的可读反馈；
- 重连后的完整权威快照和未读战斗摘要。

具体前端框架、组件库、主题和视觉品牌不属于本契约。按钮或快捷操作可以存在，但必须映射到与文字命令相同的服务端意图，不能在客户端直接改变游戏状态。

### 5.3 必须可达的玩家意图

语法和别名可以由命令协议细化，但玩家至少能够表达：

```text
look / exits / go
talk / ask / quest / accept / abandon / turn-in
attack / cast / flee / status
loot / decline-loot / inventory / equip / use
buy / sell
train / help
recover
say（可选）
```

### 5.4 多人和持久化

- 两个独立连接可以绑定两个不同角色并进入同一房间；服务端至少支持 5 个并发客户端连接，以满足 [07 §8](./07-acceptance-and-playtest.md) 的 3–5 人测试组合，战斗内玩家上限仍为两名；
- 两名玩家可以进入同一个 `CombatSession`，接收同一权威事件顺序；
- 每个角色只有一个逻辑权威 `CharacterState`；在线时由单一 live 实例承载，PostgreSQL 只保存其最近耐久快照。角色受 active 战斗控制时，仅所属 session 的唯一运行时投影临时覆盖当前生命／资源。该投影不能复制或整体反写成长／资产，奖励账本也只记录幂等结果和待领取个人掉落；本条为产品级冻结范围声明，live／durable 与投影的规则正文见 [01 §2](./01-player-flow.md)，通用工程机制见[多人一致性 §9](../multiplayer-consistency.md)；
- `CharacterState` 的全部字段／子状态按 [01 §2](./01-player-flow.md) 持久化，`slice_complete` 由 `sliceCompleteAt` 派生；
- 非战斗重连恢复位置和角色状态；
- 战斗中断线由 `afk_profile` 继续，重连恢复继续演算后的状态而不是断线瞬间；
- 持久奖励先提交成功，再向客户端显示永久成功结果。

## 6. Out of scope

- 艾尔文森林主体、闪金镇内部、暴风城、西部荒野、死亡矿井和其他地区；
- 盗贼、牧师、圣骑士、术士及其他种族；
- 正式组队、邀请、队长、队伍战术、匹配、五人副本和团队系统；
- CompanionController、AI 队友和任何生成式 AI 文字表现；
- 共享掉落、抢拾取、需求／贪婪、交易和邮件转移；
- 完整跑尸、灵魂状态、灵魂医者、耐久损失和复活虚弱；
- 持久聊天历史、离线消息、公会、好友、屏蔽和聊天审核后台；
- 公开注册、密码找回、多角色管理和完整账号生命周期；
- 前端框架承诺、桌面安装包、Tauri、Telnet 和移动原生客户端；
- PvP、完整仇恨／组队职责、AI 队友、完整职业扩展和高等级平衡；
- 生成式任务、动态剧情或由 AI 裁决规则；
- 公共发行、付费、运营和品牌体系。

## 7. Definition of Done

本节只陈述产品级完成条件；量化门槛、边界样本与试玩记录要求见 [07 §11](./07-acceptance-and-playtest.md)。

### 7.1 端到端玩家闭环

- [ ] 使用预置测试账号时，玩家可以完成“名称 + 战士／法师”的最小角色创建。
- [ ] 已有角色可以跳过创建，并从服务端权威快照恢复。
- [ ] 玩家能依靠 `look`、`exits`、任务线索和固定文本从修道院走到南路。
- [ ] 主干任务前置不能跳过，任务接取、推进、交付、奖励和后续解锁均由服务端裁决。
- [ ] 达到南路边界并满足首次前置时，`sliceCompleteAt` 只写入一次；角色仍在南路并可以向北返回。
- [ ] 完成后退出重连，`sliceCompleteAt`、派生的 `slice_complete`、位置和成长状态不丢失也不重复发放。

### 7.2 战斗、死亡与成长

- [ ] 战士与法师完成对应训练后，各自在 5 级时有 5 个当前可用动作，并呈现可观察的资源或读条差异；替换技能不会让旧等级继续进入新动作。
- [ ] 不会逃离的同级普通敌人在基准装备和推荐战术下能够完成连续自动战斗；当前敌群结束后停止，不自动攻击下一批。Young Wolf 按逃离、零奖励、遭遇关闭和刷新恢复单独验收。
- [ ] 玩家在首次开战前能查看带稳定 `ruleId` 的个人战术，在脱战时完成合法编辑或无参数重置；Worker 呼救和 Garrick 实际加入空闲 Thug 时显示关键决策帧。Garrick 强制集火在本切片冻结为服务端裁决且不可绕过；行为参数见 [04 §7.4](./04-combat-and-progression.md)。
- [ ] 玩家能够提交一次性覆盖动作，并看到接受、执行、失效或被替换的结果。
- [ ] 每个 `(SpawnInstanceId, RewardEpoch)` 的经验、任务进度、任务物品和个人掉落在同一权威结算单元中形成，不因重试重复产生；多敌遭遇不被误当作一个跨敌人事务。
- [ ] 死亡后只有合法恢复命令可改变角色；`recover` 将角色恢复到修道院庭院，且不会丢失已提交进度。
- [ ] 两名玩家全部死亡时遭遇可重置；一人死亡时不会强制终止仍有存活参与者的战斗。

### 7.3 两人共享状态与个人掉落

- [ ] 两个客户端能同时看到彼此进入、离开和断线状态。
- [ ] 两名玩家可以不建立正式队伍而进入同一个 `CombatSession`。
- [ ] 同一场战斗的关键事件顺序和胜负对两个客户端一致。
- [ ] 每名符合资格的参与者只看到并领取自己的个人掉落；任一方领取不会消耗另一方的结果。
- [ ] 两名玩家的任务／知识、经验／等级、生命／资源与死亡状态、金币、普通／任务物品栏、装备、已训练动作、个人掉落和 `sliceCompleteAt` 不互相覆盖。

### 7.4 断线与恢复

- [ ] 角色在战斗中断线后，`queued` 手动覆盖无成本取消、`started` 动作继续且恰好结算一次，随后切换到固定 `afk_profile` 完成当前遭遇。
- [ ] 断线角色在当前遭遇结束后停止，不移动、不跟随另一名玩家、不攻击刷新敌人。
- [ ] 重连时旧连接失去控制权，客户端获得当前权威快照和断线期间摘要。
- [ ] 重连清除 `disconnect` 原因并恢复 `active_profile`；已经消费的命令不会再次执行。
- [ ] 若角色已经死亡，重连后显示死亡状态和 `recover`，而不是自动复活。

### 7.5 表现、测试与范围控制

- [ ] 所有玩家可见文本来自固定模板或结构化内容，运行路径中没有生成式 AI 调用。
- [ ] 最小 Web 终端在不依赖特定前端框架语义的情况下完成输入、反馈、快照和重连。
- [ ] 关键领域测试使用可控时钟和固定随机种子，覆盖个人战术、关键决策帧、单人战斗、两人同战斗、个人掉落、死亡、战斗中断线和 `sliceCompleteAt` 幂等。
- [ ] 至少完成一次真实的目标路径试玩，并记录迷路点、命令错误、死亡、断线与恢复结果。
- [ ] `say` 未实现时不阻塞完成；若实现，房间广播有效且数据库、重连快照与未读摘要中均不存在聊天历史。
- [ ] Out of scope 能力没有以“先搭接口”或“以后会用”为理由进入生产路径。

## 8. 非阻塞实现选择

以下选择可以在实现时确定，但不得改变本契约的玩家语义：

- Web 前端框架、组件库和样式方案；
- 名称的具体字符集与长度上限，但必须由服务端校验并提供可恢复错误；
- `say` 是否随北郡 v1 一同交付；
- 命令别名、快捷键和纯表现布局；但不得暴露或把文字 `retreat` 映射为 `flee`，也不得增加 [05 §3–§4](./05-tactics-and-decisions.md) 定义之外的战术编辑操作、任何 afk 表编辑意图，或绕过 04／05 的固定命令集；
- 固定模板的具体文案，只要不改变任务事实和状态结果；
- Node.js 接入库、Schema 库和测试运行器的具体选择。

任何会改变个人掉落、死亡恢复、断线接管、双人同战斗或 `slice.complete` 语义的变更，都必须先更新本文和对应玩家流程。
