## 0. 设计结论

战斗采用：

> **连续自动执行、约 5 秒战斗摘要、持续战术、关键时刻可覆盖；距离只表示角色之间的接战关系。**

核心原则：

1. 普通战斗不要求玩家持续输入，角色依靠已保存战术完成当前遭遇。
2. 玩家操作用于提高效率、节约资源和处理异常，而不是维持角色运转。
3. 当前敌群结束后立即停止；不自动选择下一批敌人、不追击到其他房间。
4. 重要机制从公开意图到结算至少保留 5 秒，不设置更短的必答反应。
5. 玩家、暂离玩家与规则型队友控制器使用同一套技能、资源、仇恨、接战和遭遇规则。
6. 首版不模拟地形、战斗区域、精确码数、朝向、遮挡或视线。
7. 保留的是 WoW 的职业资源、读条与打断、仇恨、治疗、控制、装备成长和五人协作，而不是它的操作频率。

### 0.1 设计目标

- 使用基准装备和推荐战术时，同级普通敌人可以在零额外输入下稳定完成。
- 玩家离开几十秒至数分钟后，能够从一份摘要中理解发生了什么。
- 主动接管战斗有明显价值，但普通战斗不会因为玩家没有盯屏而停摆。
- 四职业即使在自动执行时，也具有不同的资源节奏和战斗职责。
- 死亡矿井能够验证坦克、治疗、输出、控制、打断和仇恨抢救。

### 0.2 非目标

首版不追求：

- 精确复刻 1.12 的秒级操作、网络延迟和法术批处理；
- 精确八码、三十码或四十码距离；
- 地板技能、扇形、背后角度和二维战斗地图；
- 可编程的挂机脚本或通用条件表达式（固定词表、固定槽位内的战术表构筑不在此列，见第 3 节）；
- PvP 控制递减、竞技场平衡和复杂抗性分段；
- 玩家完全不做准备也能自动击败精英或 Boss。

## 1. 玩家注意力契约

“可暂离”必须是明确规则，而不是模糊承诺。

### 1.1 系统保证

玩家只是停止输入时继续使用 active_profile；执行“afk on”或网络断线后改用 afk_profile。两种情况都遵守以下保证：

- 当前遭遇继续执行；
- 无效动作自动回落到合法的基础动作，不会整轮空转；
- 当前遭遇结束后停止战斗并进入恢复或等待；
- 不主动攻击新目标；
- 不追击逃入其他房间的敌人；
- 暂离期间的普通攻击和小额治疗被压缩，不在返回时刷出完整日志；
- 返回时显示战斗结果、关键机制、资源消耗、掉落与死亡原因。

系统只保证角色不会因为“没有输入”而停止运转，**不保证任何战斗都绝对安全**。

### 1.2 内容的注意力等级

| 内容 | 默认战术的预期表现 | 玩家是否需要接管 |
|---|---|---|
| 同级普通怪 | 基准装备与推荐战术下应能稳定完成 | 可选；主要用于提速和省资源 |
| 两只普通怪 | 通常能完成，但消耗明显增加 | 建议关注 |
| 三只普通怪 | 可能控制、撤退或失败 | 建议接管 |
| 精英怪 | 可以按阈值撤退，不承诺取胜 | 应主动准备 |
| 副本杂兵 | 按标记、控制和职责执行 | 队长需在战前配置 |
| Boss | 战术继续运行，但可能正常团灭 | 开战前必须确认和分工 |

### 1.3 停止条件

自动执行只覆盖当前遭遇。出现以下任一情况即停止：

- 敌方全部死亡或逃离；
- 玩家成功撤出遭遇；
- 玩家一方全部死亡；
- Boss 脱战并重置；
- 脚本明确结束遭遇。

系统不会自动跑尸、自动复活或再次开怪。

### 1.4 刷新与静止玩家

为了让短暂离开成为可靠行为：

- 敌人刷新不会仅因玩家静止在房间中而自动开启新遭遇；
- 常规敌对行为在玩家进入房间、主动接触、攻击或触发明确事件时发生；
- 剧情伏击、巡逻和高危区域可以例外，但必须在房间描述中明确提示；
- 这一规则不能被用于自动刷经验，因为战斗结束后不会自动选取刷新目标。

### 1.5 消耗品权限

自动战术不得擅自消耗稀有资源。角色分别保存：

| 权限 | 行为 |
|---|---|
| 禁用 | 不自动使用药水、绷带、食物或长冷却技能 |
| 仅保命 | 达到撤退或濒死阈值时允许使用指定物品 |
| 允许使用 | 按战术阈值使用白名单中的物品 |

默认值为“禁用”。后续共享战利品系统也不得替暂离玩家擅自选择需求；只有后续规格明确启用时，才可按预设拾取策略选择放弃或贪婪。北郡 v1 使用 PersonalLoot，断线／AFK 不会自动领取、放弃或改变其状态，掉落保持 pending，见[v1 多人与恢复规格](./northshire-v1/06-multiplayer-and-recovery.md)。

### 1.6 主动暂离与断线

“afk on”执行以下操作：

- 当前遭遇改用 afk_profile；没有自定义时，使用保留原职责的保守预设；
- 不再接受需要即时确认的非必要动作；
- 不自动使用未授权消耗品或长冷却；
- 当前遭遇结束后停止并进入等待；
- 不自动跟随队伍进入下一场战斗。

“afk off”恢复 active_profile 并重新接受输入，但不会自动开怪。

意外断线与主动暂离使用同一套当前遭遇接管规则（北郡 v1 只实现断线来源，手动暂离命令留给后续阶段）。战斗结束后，角色进入等待并执行服务器的常规安全退出流程；不会因重连重复执行命令。多人 Boss 战不会因为一名玩家断线而暂停，该角色仍可能死亡，队伍也可能正常团灭。

运行时 afk 状态记录 reason：manual 或 disconnect。成功重连会自动清除 disconnect 并恢复 active_profile；玩家主动设置的 manual 状态必须通过“afk off”解除。

## 2. 时间模型：连续模拟，低频决策

### 2.1 两种时间尺度

战斗同时存在两个时间尺度：

1. **内部模拟时间**  
   处理武器攻速、约 1.5 秒公共冷却、施法时间、引导、持续效果、盗贼能量跳动和冷却。

2. **战斗摘要帧**
   默认约 5 秒汇总一次状态、危险与计划。它是输入和输出节奏，不是传统意义上的固定回合。

因此，约 5 秒内可以发生多次普通攻击或若干自动技能；玩家不需要逐个 GCD 下令。

### 2.2 动作来源与优先级

每个合法技能槽按以下优先级选择动作：

1. 眩晕、恐惧等强制状态；
2. 玩家提交的一次性覆盖命令；
3. 玩家授权的紧急反应；
4. 持久战术选择；
5. 基础攻击或安全等待。

一次性命令只覆盖下一个合法动作，不会暗中修改长期战术。

若目标死亡、资源不足、技能冷却或条件不再成立：

- 取消该覆盖命令；
- 输出一条简短原因；
- 立即回落到持久战术；
- 不让角色因陈旧命令空转。

### 2.3 指令队列

- 每名角色只有一个普通覆盖槽。
- 新的普通命令替换尚未执行的旧命令。
- 打断、嘲讽和保命可以进入单独的优先反应槽。
- 命令在下一个合法执行点生效，而不是因为还差少量 GCD 就失败。
- 命令被接受只表示已经进入或替换覆盖槽，不表示技能已经结算；执行、失效或被替换分别产生后续事件。
- 重连不会重复执行断线前已经消费的命令。

### 2.4 关键机制预告与截止时间

Boss 或精英的关键机制必须保存 announced_at 与 resolves_at，并满足 resolves_at - announced_at 不少于 5 秒。战斗摘要帧只负责汇报，技能不会因为显示帧提前或延后结算：

~~~text
[第 8 战斗摘要]
拉克佐正在蓄力“碎骨重击”，预计约 5 秒后命中当前目标。

默认反应：战士使用盾牌格挡；牧师保留快速治疗。
你可以覆盖：taunt / shield block / flash heal / continue
~~~

无人输入时执行已经配置的默认反应。未配置、技能不可用或资源不足时，机制正常命中。

这类预告向玩家呈现为**关键决策帧**：编号选项菜单加倒计时，每个选项映射到一条既有覆盖或反应命令，数字快捷输入即提交该命令。它是既有命令之上的呈现层，不是新的战斗状态；普通遭遇每场至多出现少数几帧，玩家全程不选也只是回到持久战术与默认反应。北郡 v1 的最小实现见 [v1 战术构筑与关键决策规格](./northshire-v1/05-tactics-and-decisions.md)。

### 2.5 单人与多人

- 多人遭遇不会因为单个玩家离开而暂停。
- 暂离玩家由 afk_profile 接管；仍在线但暂时没有输入的玩家继续使用 active_profile。
- 私人单人实例可以提供可选的“危险时暂停”，但它不是多人规则，也不是首版必需能力。
- Boss 开战前使用准备确认；开始后不保证暂离玩家所在队伍能够获胜。

## 3. 持久战术

持久战术的目标是让角色在没有逐秒命令时仍然表现得像该职业，而不是提供一门挂机编程语言。

### 3.1 通用战术字段

| 字段 | 可选值示例 | 作用 |
|---|---|---|
| 姿态 | 进攻／均衡／保守 | 决定伤害、保命和资源倾向 |
| 选敌 | 协助坦克／标记优先／指定目标 | 决定普通技能目标 |
| 资源 | 爆发／正常／节省 | 决定资源保留和高消耗技能 |
| 反应 | 打断／自保／保护治疗者／驱散 | 决定优先反应顺序 |
| 撤退 | 生命阈值／治疗死亡／队伍人数 | 决定何时尝试退出 |
| 消耗品 | 禁用／仅保命／白名单 | 限制自动消耗 |
| 范围技能 | 禁用／安全时使用／允许 | 防止破坏控制或无谓消耗资源 |

### 3.2 模板与受限构筑，而非脚本

战术是玩家的核心策略载体：每名角色从职业模板初始化一份个人战术，可以在**固定词表与固定槽位内**编辑——开关规则行、调整求值顺序、在枚举集中选阈值。保命行与基础攻击回退行固定存在，保证暂离与断线时角色不失去底线行为。首版模板包括：

- 单人均衡；
- 单人保守；
- 坦克守仇恨；
- 输出协助坦克；
- 治疗节省法力；
- Boss 保留冷却。

玩家不能编写任意条件、循环和脚本；构筑深度通过成长解锁扩展——训练师和任务奖励提供新的条件行与槽位，让战术构筑成为与装备平行的成长线。战术复杂度应随着职业能力增加，而不是先建设一个通用规则引擎。北郡 v1 的最小可编辑集（行开关、排序、枚举阈值）由 [v1 05 规格](./northshire-v1/05-tactics-and-decisions.md) 冻结。

### 3.3 手动覆盖示例

~~~text
tactic conservative
focus 迪菲亚监工
reserve interrupt
cast polymorph on 迪菲亚矿工#2
taunt 迪菲亚监工
retreat
flee east
~~~

其中“tactic”和“focus”是持久修改，其余通常只覆盖下一次合法行动。

## 4. 遭遇边界

### 4.0 术语

| 术语 | 含义 |
|---|---|
| WorldInstanceId | 世界执行、广播和串行化作用域，可包含多个房间和多场战斗 |
| RoomId | 文字世界中的位置 |
| EncounterDefinitionId | 可选的静态敌群或 Boss 脚本定义；普通野外战斗可为空 |
| CombatSessionId | 一次运行中的实际战斗 |
| SpawnInstanceId | 一次实际生成的敌人身份 |
| RewardEpoch | 该生成实例当前可发放奖励的唯一代次 |
| DecisionFrameNo | 约 5 秒一次的表现层汇报编号 |

WorldInstanceId 是单写者队列；CombatSessionId 标识战斗状态，但不另设并发写者。后续若启用两场战斗合并，也必须在同一个世界实例队列中完成。

### 4.1 遭遇不是房间

房间负责世界位置，遭遇负责战斗边界：

- 同一房间可以存在两个互不相关的遭遇；
- 每个角色同一时刻最多属于一个遭遇；
- `[later]` 对其他遭遇中的目标造成攻击、治疗或增益时，两场遭遇按后续规则合并；
- 仇恨、治疗仇恨、范围技能、掉落归属和战斗结束均以遭遇为边界；
- 不同房间默认不能互相攻击或治疗。

以上合并规则只面向后续阶段。北郡 v1 不合并 CombatSession：已经在一场战斗中的角色不能加入另一场，同房间两场无关战斗之间的攻击、治疗或增益均被稳定拒绝，也不会产生仇恨或资格变化。

### 4.2 加入与合并

以下行为可使角色加入遭遇：

- 主动攻击其中的参与者；
- 治疗或增益其中的参与者；
- 被敌人选为目标；
- 触发敌群协助或遭遇脚本；
- 接受队伍明确的参战命令。

遭遇合并后，仇恨表不会凭空复制；每个敌人只记录实际产生过威胁的角色。

### 4.3 参与者与旁观者

- 遭遇参与者接收完整战斗事件与决策摘要；
- 同房间但未参战的旁观者只接收开战、加入、死亡、逃跑和结束等简要事件；
- 旁观本身不会产生仇恨，也不会自动加入遭遇；
- 旁观者攻击、治疗或增益参与者后，后续阶段可按加入与合并规则处理。
- 北郡 v1 例外：按[多人与恢复规格](./northshire-v1/06-multiplayer-and-recovery.md)，共享敌人的生命档位与公开意图向同房间所有玩家公开；旁观者跨 CombatSession 的攻击、治疗或增益仍被拒绝，本节其余旁观与合并规则适用于后续阶段。

## 5. 距离模型：只有接战关系

首版不做战斗地图，也不把房间拆成前排、后排或命名区域。

### 5.1 三种运行时状态

距离只用三个关系状态表达：

- **engaged_with**：已经与哪些敌人接战；
- **approaching**：正在接近哪个目标；
- **withdrawing**：正在尝试脱离哪些目标。

“A 与 B 接战”是对称关系，但不代表双方正在攻击同一个目标。一个角色可以同时与多个敌人接战。

### 5.2 基本规则

- 近战技能要求使用者与目标已经接战。
- 同一遭遇中的远程技能、治疗和增益默认可以选择任意合法目标。
- 普通接近的基础耗时为 5 秒；完成后建立接战关系。
- 冲锋等技能跳过普通接近直接建立接战，可带有技能自身定义的短延迟，并附带自身效果；北郡 v1 的准确时长只由其[战斗与成长规格](./northshire-v1/04-combat-and-progression.md)定义。
- 普通脱离的基础耗时为 5 秒，并可能被追击阻止。
- 闪现、击退、恐惧等效果可以直接解除或改变接战关系。
- 不同房间不能接战。

### 5.3 攻击目标、仇恨目标与接战关系

三者必须分开：

- **攻击目标**：角色当前准备使用技能的对象；
- **仇恨目标**：敌人根据仇恨表决定追击或攻击的对象；
- **接战关系**：哪些角色已经进入近战接触。

例如，战士、盗贼和监工可以彼此接战，但监工当前攻击战士；盗贼因此满足背刺的抽象条件。

### 5.4 接近

当近战敌人选择一个尚未接战的目标：

~~~text
迪菲亚监工放弃战士，开始接近牧师。
若未被阻止，将在约 5 秒后与牧师接战。
~~~

此时队伍可以：

- 战士嘲讽；
- 盗贼控制；
- 法师变形或使用其他远程控制；
- 牧师恐惧；
- 或者接受目标转移。

远程敌人不需要先建立接战即可攻击同一遭遇中的目标。

### 5.5 脱离与防止无限风筝

普通“retreat”不能免费摆脱所有近战敌人：

- 未受减速、定身、昏迷或其他控制的追击者，可以阻止普通脱离；
- 只有明确标记为可阻止追击的减速或控制、闪现、击退，或者敌人主动放弃追击时，才能可靠制造远程空档；
- 结算时逐条检查接战关系：被定身、昏迷、恐惧或有效减速的敌人无法阻止这次脱离；至少一名未受限制且选择追击的敌人可以维持自己与撤退者的接战关系；
- 撤退成功与否不要求所有接战关系得到同一结果，角色可以摆脱部分敌人但仍被另一名敌人缠住；
- 已被定身的敌人如果已经接战，仍可攻击接战目标，但不能追赶成功脱离的目标；
- 减速是否能阻止本次追击由效果字段 blocks_pursuit 声明，不比较隐藏的移动速度百分比；
- “retreat”只改变战斗内接战关系；
- “flee”通过房间出口退出整个遭遇，两者不能混用。

### 5.6 施法与接战

被接战不等于不能施法：

- 火球、治疗等法术仍可尝试施放；
- 近战伤害可以造成施法延迟，首版每次施法最多延迟一次；
- 明确的打断技能会取消施法，并按技能规则造成法术系封锁；
- 护盾或天赋可以减轻施法延迟；
- 魔杖及未来远程武器通过 usable_while_engaged 明确声明近身限制。

### 5.7 背后、击退和冲锋

- 背刺不判断朝向；条件改为“与目标接战，且目标当前没有攻击盗贼”。
- 击退解除指定接战关系，并可附加短暂失衡。
- 冲锋对未接战目标使用，跳过普通接近，按技能定义的延迟建立接战。
- 冰霜新星影响与法师接战或正在接近法师的敌人，并阻止其追击；它不会凭空解除已经存在的接战。
- 闪现可以解除法师身上的接战关系。

### 5.8 范围目标

范围技能只使用明确的目标集合，不使用地面位置或“目标附近”：

| 目标类型 | 含义 |
|---|---|
| self | 自己 |
| single_enemy | 单个敌人 |
| single_ally | 单个队友 |
| engaged_enemies | 与施法者接战的敌人 |
| engaged_or_approaching_self | 与施法者接战或正在接近施法者的敌人 |
| encounter_enemies | 当前遭遇中的敌人，可设置数量上限 |
| party | 当前遭遇中的队友 |

例如：

- 顺劈斩：最多命中两个 engaged_enemies；
- 冰霜新星：影响所有 engaged_or_approaching_self；
- 暴风雪：影响最多五个 encounter_enemies；
- 治疗祷言：影响 party；
- 不存在“选择一个地面区域”的操作。

## 6. 仇恨与目标切换

仇恨是必须保留的 WoW 核心系统，不能用“Boss 固定攻击战士”替代。

### 6.1 基础威胁

初始调参基线：

- 有效伤害约产生 1:1 威胁；
- 有效治疗产生较低威胁，并分摊给当前相关敌人；
- 坦克姿态、破甲、嘲讽等技能具有倍率或固定威胁；
- 过量治疗不产生威胁；
- 控制与增益是否产生威胁由技能明确声明。

精确数值属于平衡参数，不属于不可变历史事实。

### 6.2 切换门槛

为防止文字输出中的目标频繁抖动，可采用：

- 已与敌人接战的竞争者达到当前目标约 110% 威胁时抢走目标；
- 尚未接战的竞争者达到约 130% 威胁时使敌人转向并开始接近；
- 这些百分比是首版调参起点。

### 6.3 嘲讽

嘲讽：

- 在短时间内强制敌人选择使用者；
- 将使用者提升到当前最高威胁梯队；
- 如果敌人正在接近其他目标，取消该接近并重新选择；
- Boss 是否免疫由遭遇数据明确声明，不能默认全部免疫。

### 6.4 玩家看到的信息

默认界面不显示精确威胁数字，只显示：

- 稳固；
- 逼近；
- 危险；
- 已转移。

调试命令可以显示完整仇恨表。

## 7. 职业节奏

职业差异必须来自资源、自动循环和主动决策，而不是更换技能名称后使用同一套行为。

### 7.1 战士

自动执行：

- 按武器速度普通攻击；
- 造成和承受伤害时获得怒气；
- 按姿态维持基础伤害或基础威胁。

玩家或战术决定：

- 是否保留怒气给嘲讽后的抢仇恨；
- 何时使用破甲、盾牌格挡和打断；
- 是否冲锋建立接战；
- 何时从输出转为保守。

辨识点：越危险时怒气越充足，资源和风险正相关。

### 7.2 盗贼

自动执行：

- 按内部节奏恢复能量；
- 使用基础攻击积累连击点；
- 协助坦克攻击标记目标。

玩家或战术决定：

- 为脚踢保留多少能量；
- 何时使用终结技；
- 控制哪个目标；
- 是否使用闪避或脱离；
- 是否允许范围攻击。

辨识点：等待能量、安排连击点，并在输出与控制之间取舍。

### 7.3 法师

自动执行：

- 按爆发、正常或节省策略选择基础法术；
- 保持指定控制目标；
- 在安全条件满足时使用允许的范围技能。

玩家或战术决定：

- 何时变羊、反制或冰环；
- 是否保留法力；
- 被接战后继续施法、控制还是尝试脱离；
- 何时使用长冷却爆发。

辨识点：读条、控制、法力与被近战缠住后的风险。

### 7.4 牧师

自动执行：

- 按治疗阈值选择高效治疗；
- 在无人需要治疗时使用低消耗伤害或魔杖；
- 按授权处理可驱散状态。

玩家或战术决定：

- 紧急治疗阈值；
- 何时使用护盾、快速治疗和驱散；
- 为 Boss 后半段保留多少法力；
- 是否为了救人接受更高治疗仇恨。

辨识点：治疗优先级、法力规划和五秒回蓝。

### 7.5 五秒回蓝与职业计时

- 武器攻速、公共冷却、能量跳和法术读条在内部时间中运行。
- 法师与牧师停止消耗法力约五秒后进入正常回蓝。
- 战斗摘要帧只汇报结果，不改变职业资源的内部节奏。
- 不要求玩家按秒输入才能利用这些机制。

## 8. 状态、控制与打断

所有持续效果统一表示为 aura，至少包含：

- 来源与目标；
- 持续时间；
- 叠层；
- 周期效果；
- 控制类型；
- 可驱散类型；
- 对技能、接战和仇恨的修正。

首版需要：

- 持续伤害与持续治疗；
- 护盾；
- 眩晕；
- 恐惧；
- 定身；
- 变形；
- 减速；
- 增益与减益；
- 打断与短暂法术系封锁。

首版不做 PvP 控制递减、复杂抗性分段和 Boss 通用控制免疫表。每个 Boss 的免疫由遭遇数据明确声明。

## 9. 五人队与规则型队友控制器

本节的 CompanionController 是确定性规则控制器，不是生成式叙事 AI。生成式模型只能负责文字表现，不能选择技能或裁决战斗。

### 9.1 队伍分工

五人上限、手工邀请和亲自前往副本入口继续保留。低等级不硬编码现代专精，但自然形成：

- 坦克与仇恨；
- 治疗与法力；
- 输出与资源；
- 控制；
- 打断；
- 标记与集火。

### 9.2 队长命令

队长只下达粗粒度战术：

~~~text
mark skull 迪菲亚监工
mark moon 迪菲亚巫师
order mage control moon
order rogue reserve interrupt
order priest conserve 30
assist tank
ready
~~~

玩家不需要逐个 GCD 遥控四名规则型队友。

### 9.3 CompanionController 边界

规则型队友控制器：

- 使用与真人完全相同的技能、资源、冷却、读条、接战和仇恨；
- 只能响应玩家也能看见的结构化意图；
- 不读取隐藏 Boss 阶段；
- 不提前知道随机目标；
- 只在意图公开后经过 automation_reaction_delay 才能提交反应；首版默认 1 秒且不得为零，并与真人共用同一个 resolves_at 截止时间；
- 不拥有无限法力；
- 不跳过接近、脱离或控制过程。

暂离真人使用单独保存的 afk_profile，行为规则与 CompanionController 相同。afk_profile 默认保留原职责，例如坦克仍优先守仇恨、治疗仍优先保命，但采用更保守的资源和消耗品权限。

### 9.4 少人队伍

优先由规则型队友补齐队伍职责。1–3 名真人时可以适度调整生命与伤害，但不能因此删除：

- 仇恨；
- 治疗；
- 控制；
- 打断；
- 接战；
- Boss 阶段。

## 10. 敌人与 Boss

### 10.1 机制预算

| 敌人类型 | 机制预算 |
|---|---|
| 普通怪 | 最多一个辨识机制 |
| 精英怪 | 一至两个可读机制 |
| Boss | 一个主要应对机制、一个阶段变化、一个资源压力点 |

禁止使用：

- 从公开意图到结算不足 5 秒的必答机制；
- 地板安全区；
- 精确站位；
- 遮挡或视线；
- 背后角度；
- 大量同时刷出的微事件。

“不少于 5 秒”约束的是 Boss 与精英需要玩家响应的必答机制。普通怪的单一辨识机制默认由持久战术自动处理，不属于必答机制，其公开窗口可以短于 5 秒；北郡 v1 的准确窗口只由其[战斗与成长规格](./northshire-v1/04-combat-and-progression.md)定义。

### 10.2 适合文字战斗的机制

- 至少提前 5 秒公开的重击；
- 可打断施法；
- 可驱散状态；
- 增援敌人；
- 控制分工；
- 仇恨转移；
- 接近治疗者；
- 击退导致解除接战；
- 生命阈值阶段转换；
- 法力与长冷却规划。

### 10.3 死亡矿井改编

| Boss | 文字化机制 |
|---|---|
| 拉克佐 | 至少提前 5 秒预告重击；坦克防御或治疗预判 |
| 斯尼德的伐木机／斯尼德 | 载具与驾驶者两阶段，第二阶段改变目标策略 |
| 基尔尼格 | 可打断的熔炉过载；失败后全队承受灼烧，不要求移动 |
| 重拳先生 | 约 66%／33% 生命切换武器与攻击模式 |
| 绿皮队长 | 击退解除坦克接战，随后可能转向高仇恨远程 |
| 范克里夫 | 护卫增援优先接近治疗者，需要控制、嘲讽或集火 |

每个机制都必须同时定义：

- 玩家可见的意图；
- announced_at、resolves_at 与不少于 5 秒的响应窗口；
- 默认战术是否能响应；
- 手动覆盖方式；
- 无人响应时的正常后果；
- CompanionController 可见的信息边界。

## 11. 输出设计

### 11.1 每个战斗摘要帧回答四个问题

1. 发生了什么重要变化？
2. 当前最大的危险是什么？
3. 自己和队伍还剩多少关键资源？
4. 持久战术下一步准备做什么？

### 11.2 输出示例

~~~text
[第 6 战斗摘要｜范克里夫 49%]

两名迪菲亚护卫加入战斗。
护卫#1 正在接近牧师；护卫#2 已与战士接战。

仇恨：范克里夫→战士（稳固）；护卫#1→牧师（危险）
资源：战士怒气 42；牧师法力 37%；盗贼脚踢可用

默认计划：战士嘲讽护卫#1；法师控制护卫#2；全队继续集火范克里夫。
> taunt 护卫#1
[已覆盖下一合法动作：嘲讽 护卫#1]
~~~

### 11.3 暂离摘要

玩家返回时只显示：

- 战斗胜负；
- 经过多少战斗摘要帧；
- 关键目标切换和机制；
- 生命、法力与消耗品变化；
- 获得的经验、任务进度与掉落；
- 撤退或死亡原因。

完整逐击日志可以保存在短期调试记录中，但不会自动倾倒给玩家。

### 11.4 最小命令

~~~text
attack / cast / use / flee
status / summary / combatlog
tactic / focus / assist
taunt / interrupt
approach / retreat
afk
~~~

技能原名和简短别名都映射到结构化动作，命令层不直接修改生命或仇恨。

## 12. 结算顺序

命令与计时器持续进入权威实例的串行队列；战斗摘要帧只是一个约每 5 秒触发的 `combat_summary_due` 汇总事件，不开启或关闭输入窗口，也不同于低频出现、带编号选项的关键决策帧。

内部事件按单调时钟时间戳执行：

1. 接收命令或到期计时器，并在串行队列中重新验证状态；
2. 按“强制状态 → 玩家覆盖 → 授权反应 → 持久战术 → 基础动作”选择下一合法动作；
3. 执行到期的普通攻击、技能、读条、持续效果和资源变化；
4. 处理接战 transition、仇恨目标、死亡和遭遇阶段；
5. 记录结构化事件；
6. 战斗摘要帧到期时，聚合自上次汇报以来的事件并生成摘要。

关键敌人意图在产生时立即广播，并携带 announced_at 与 resolves_at；它不等待下一个战斗摘要帧才开始计时。

外部命令到达权威服务端时获得单调时钟 ingressAt 与递增 ingressSeq。处理 ingressAt=T 的外部命令前，WorldInstance 队列必须先结算全部 dueAt<=T 的内部事件；dueAt 与 ingressAt 相等时内部事件优先，因此恰好在截止时间到达的打断视为过晚。随后，同 ingressAt 的外部命令按 ingressSeq 执行，内部事件按 `(dueAt, priority, eventSeq)` 稳定排序。

Node.js 定时器回调只负责唤醒队列，不能决定规则顺序。即使到期回调延迟，队列仍按原始 dueAt 补结算，不允许更晚到达的命令抢在已经到期的事件之前。不使用客户端时间戳或数据库提交先后来决定胜负。

## 13. 数据模型

现有内容实体继续使用 zone、room、npc、mob、quest、item 和 ability；只有脚本化敌群或 Boss 额外使用 encounter。本节增加运行时战斗状态。

### 13.1 运行时实体

| 实体 | 主要字段 |
|---|---|
| combat_session | WorldInstanceId、CombatSessionId、可空 EncounterDefinitionId、房间、参与者、开始时间、最近汇报时间、遭遇阶段、结束原因；普通野外战斗由实际 SpawnInstance/mob 组成 |
| combatant_state | 当前 Attempt 的生命／资源运行时投影、目标、active_profile、afk_profile、当前 afk 状态及 reason、覆盖动作、读条、状态；不是第二份角色权威余额 |
| tactics_profile | 姿态、选敌、资源、反应、撤退、消耗品与范围技能权限 |
| engagement | 对称的两名参与者、建立方式、开始时间 |
| engagement_transition | 接近／脱离、发起者、相关目标、started_at、completes_at、状态与取消原因 |
| threat_entry | 敌人、威胁来源、当前值与修正 |
| declared_intent | 敌人公开意图、announced_at、resolves_at、可用响应 |
| aura | 来源、目标、持续、叠层、周期、控制与驱散类型 |
| combat_summary | 胜负、关键事件、资源变化、奖励与失败原因 |
| defeat_reward_record | WorldInstanceId、SpawnInstanceId、RewardEpoch、characterId、rewardKind、rewardSlot 与唯一领取状态 |

### 13.2 ability 字段

技能至少定义：

- 资源消耗；
- 内部施法时间；
- 公共冷却与独立冷却；
- 目标类型；
- 是否要求接战；
- 是否建立或解除接战；
- usable_while_engaged；
- blocks_pursuit；
- 威胁修正；
- 伤害、治疗、控制、驱散和打断效果；
- 是否允许持久战术自动使用；
- 是否属于需要显式授权的长冷却或消耗品动作；
- 文本模板。

首版不应出现：

- min_range 或 max_range 码数；
- 坐标；
- 朝向；
- 视线；
- 战斗区域；
- 地面目标点。

### 13.3 事件

战斗引擎产生进程内的类型化事件：

- damage；
- heal；
- cast_started；
- cast_completed；
- interrupted；
- intent_announced／resolved／cancelled；
- combat_summary_due；
- aura_applied／removed；
- engaged／disengaged；
- approach_started／completed／cancelled；
- withdraw_started／completed／failed；
- threat_target_changed；
- defeated；
- reward_committed；
- death；
- fled。

任务图可以消费 defeated，遭遇可以消费生命阈值，输出层可以聚合事件。这里不需要基础设施事件总线。

### 13.4 内存状态、持久状态与恢复

数据库持久化：

- active_profile 与 afk_profile；
- 角色长期资源、生命和其他检查点状态；
- 消耗品权限与拾取策略；
- 已提交的经验、任务进度、掉落和战斗结果；
- SpawnInstanceId、RewardEpoch、终结原因、可信墙钟 `respawnEligibleAt`、已击败状态与 defeat_reward_record；
- 最近安全位置、不可变的 `CombatStartCheckpoint`、独立战斗 lifecycle 及最近一场未读或最终战斗摘要。

应用进程内保存：

- 当前遭遇、参与者与仇恨表；
- 短冷却、读条、aura；
- engagement 与 engagement_transition；
- 玩家覆盖命令和未结算意图。
- 当前 afk 标志；该标志是在线会话／运行时状态，不覆盖持久战术。

网络断线但服务进程仍运行时，角色按 afk_profile 继续当前遭遇；重连通过权威快照恢复显示，不重复执行已完成命令。北郡 v1 对 queued／started 手动覆盖的精确处分以其[多人与恢复规格 §6.3](./northshire-v1/06-multiplayer-and-recovery.md#63-战斗中断线)为准。

敌人死亡时的奖励提交按单一事务、唯一奖励键和不可复用的奖励代次处理，通用方向与北郡 v1 一致；具体事务边界、唯一键元组与刷新/重启语义以[多人一致性 §6 事务与不变量](./multiplayer-consistency.md)、[§7.1 战斗](./multiplayer-consistency.md)和[北郡多人与恢复规格 §4／§8](./northshire-v1/06-multiplayer-and-recovery.md)为准。

开始 Attempt 时保存一次不可变的 `CombatStartCheckpoint`，并以独立 lifecycle 记录 active、reward_pending、closed、关闭时间与原因。进程崩溃或重启时不逐事件恢复活跃战斗：清除临时战斗状态，按 lifecycle 和已经提交的击败状态重建或重置敌方遭遇，并依据对应版本规格分类恢复角色。已经终结 RewardEpoch 的敌人不能因为内存重置再次发奖；尚未提交的临时结果不得补造。北郡 v1 的准确分类以其[多人与恢复规格](./northshire-v1/06-multiplayer-and-recovery.md)为准。

## 14. 撤退、死亡与恢复

### 14.1 战斗内脱离

“retreat”尝试解除接战，但角色仍属于当前遭遇。它用于重新获得施法空间或摆脱近战纠缠。

### 14.2 逃离遭遇

“flee east”等命令通过房间出口退出遭遇：

- 需要一个明确的撤离过程；
- 可能被控制阻止；
- 成功后不自动追击敌人；
- 敌人按追击和归位规则重置；
- 队伍可以配置个人撤退或全队撤退。

### 14.3 PvE 死亡

- 不扣经验；
- 不掉落装备；
- 保留耐久和修理代价；
- 灵魂从墓地跑回尸体；
- 灵魂医者提供更昂贵的立即回归；
- 现实时间十分钟的复活虚弱改为若干场战斗或可游玩的阶段性减益；
- 全队死亡后 Boss 重置，但不会自动重新挑战。

### 14.4 战后恢复

战斗结束后：

- 停止自动攻击；
- 按配置使用普通食物、饮水或绷带；
- 稀有消耗品仍需明确授权；
- 输出一份摘要；
- 等待玩家下一条移动、攻击或任务命令。

## 15. MVP 实施顺序

> 注：本节保留的是 v1 设计包形成前的通用实施分层，不是北郡 v1 范围或验收清单。v1 已把双人同战斗与战斗中断线接管提前，并以 Garrick Padfoot 和可打断读条的 Defias Thug 取代本节的精英／进攻型施法敌人假设。北郡 v1 已明确排除本文件后续使用的战斗内五秒回蓝、玩家读条受击延迟与 `retreat`；`afk_profile` 冻结为不可编辑的内置规则，active 战术开放为脱战受限构筑并新增援军关键决策帧（见 v1 05 规格）；精确口径只以 v1 规则所有者为准。

### 15.1 通用第一阶段：早期战斗沙盒

范围：

- 战士、法师；
- 每职业 4–5 个技能；
- 三类普通敌人：近战、施法、会逃跑或呼救；
- 一个精英；
- 单人战斗。

验证：

- 自动攻击；
- 持久战术；
- 覆盖命令；
- 战斗摘要帧与关键决策帧；
- 怒气、法力和五秒回蓝；
- 读条、延迟与打断；
- 接近、接战、脱离；
- 死亡和战后停止。

### 15.2 第二阶段：职业与小队

加入：

- 盗贼、牧师；
- aura；
- 仇恨；
- 治疗威胁；
- 控制与驱散；
- 五人队；
- 规则型队友控制器；
- 暂离和断线接管。

### 15.3 第三阶段：死亡矿井

验证：

- 标记、集火与控制；
- 真实仇恨抢救；
- 增援接近治疗者；
- Boss 意图与不少于 5 秒的响应截止时间；
- 阶段变化；
- 资源规划；
- 团灭、重置和掉落。

## 16. 验收标准

### 16.1 低注意力

1. 使用基准装备和推荐 active_profile 的玩家攻击同级普通怪后停止输入两分钟，战斗能结束、角色停止且不会拉下一只。
2. 自动战斗默认不使用药水、稀有消耗品或长冷却。
3. 玩家回来只看到一份清楚摘要，而不是逐击日志。
4. 主动接管相较默认战术能明显减少时间、生命或法力消耗，但不是同级普通战的必要条件。
5. 敌人刷新不会让静止玩家自动循环战斗。
6. 模拟“奖励事务已提交、广播前崩溃”，重启后同一 SpawnInstanceId 与 RewardEpoch 不能再次产生经验、任务进度或掉落。

### 16.2 接战关系

1. 近战技能不能命中未接战目标。
2. 普通接近需要时间，冲锋跳过普通接近，按技能定义的短延迟建立接战。
3. 怪物因治疗仇恨转向牧师时先进入 approaching；战士可以用嘲讽阻止。
4. 战士可以同时与多名敌人接战，顺劈只命中技能允许的数量。
5. 法师不能仅靠反复 retreat 无限无伤风筝；配合冰环、减速或闪现后可以制造合法空档。
6. 被接战的法师仍可施法，但会承受施法延迟或打断风险。
7. 系统中不存在地形、战斗区域、精确码数或视线依赖。

### 16.3 职业与仇恨

1. 战士、法师、盗贼、牧师在自动执行时仍体现不同资源节奏。
2. 治疗产生真实威胁，不会永远安全。
3. 嘲讽、控制和打断分别解决不同问题。
4. 默认界面能明确提示目标转移，而不需要玩家查看精确数字。

### 16.4 规则型队友与 Boss

1. CompanionController 与真人在相同状态和命令下遵守相同规则。
2. CompanionController 只能响应已经公开的 Boss 意图。
3. Boss 关键机制从公开意图到结算至少保留 5 秒。
4. 未授权自动打断时，Boss 技能能够正常命中。
5. 全队战术无法承受 Boss 时允许正常团灭，但不会自动复活重试。
6. 暂离玩家不会自动对 Boss 蓝装选择需求。
7. 模拟到期计时器晚唤醒：dueAt 已到的技能必须先于更晚 ingressAt 的打断结算；时间相等时内部到期事件优先。

### 16.5 遭遇边界

1. 同一房间中的两场无关战斗互不共享仇恨和范围伤害。
2. `[later]` 明确启用跨遭遇治疗后，相关遭遇按后续规则合并。
3. 当前敌群结束后不追击、不自动选敌、不进入其他房间。
4. 北郡 v1 中，跨 CombatSession 的攻击、治疗或增益被稳定拒绝，两场战斗不会合并。

## 18. 最终判断

本项目的战斗不应是“把 WoW 日志搬进终端”，也不应退化成没有职业协作的纯数值挂机。

它的核心体验应当是：

> **战前配置职业战术，战斗中让角色持续执行；普通遭遇可以暂离，危险事件允许接管；距离只通过接战、接近和脱离表达；五人内容依靠仇恨、治疗、控制和资源形成 WoW 味道。**

只要北郡切片能够证明“零输入也能运行、主动操作仍有价值、接战关系足以承载职业差异”，才值得继续录入更大范围的任务与副本内容。
