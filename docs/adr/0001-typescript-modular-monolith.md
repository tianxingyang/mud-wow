# ADR-0001：采用 TypeScript 模块化单体权威服务端

- 状态：已接受
- 日期：2026-07-16
- 决策范围：首个可玩版本及其前期扩展

## 背景

项目目标是制作一个以 Vanilla 1.12 为资料基线、面向自己和少量朋友的文字 RPG。首个垂直切片聚焦北郡 1–5 级内容，验证移动、任务、连续自动战斗与低频决策、掉落、成长和存档恢复；后续才扩展到艾尔文森林、西部荒野与死亡矿井。

系统需要支持多个玩家同时连接，并在同一权威世界中完成房间移动、战斗和个人掉落。正式组队、竞争拾取与交易留给后续阶段，但它们同样要求统一的状态所有权与事务边界，而不天然要求微服务。当前规模也没有证明需要 Rust 的计算性能，桌面离线分发尚未被确定为首发目标。

## 决策

北郡 v1 采用 **Node.js + TypeScript 模块化单体权威服务端**：

1. 所有会改变游戏事实的命令都由服务端验证和执行。
2. 账号与会话、世界、任务、战斗、社交和持久化在代码中划分模块，但作为一个进程和一个部署单元运行。
3. Web 客户端通过 HTTP + WebSocket 命令协议连接同一个服务端；Telnet 或桌面客户端只作为后续适配器，客户端只负责输入和表现。
4. 同一房间或副本实例中的命令串行执行，避免同一份实时状态出现多个写入者。
5. 持久状态写入 PostgreSQL；奖励、拾取以及后续交易等跨记录变更使用数据库事务。
6. 核心规则由结构化数据和确定性代码裁决；北郡 v1 不调用生成式 AI，后续若启用也只能生成不改变规则结果的表现文本。

### 已确定的技术方向

- 语言与运行时：Node.js + TypeScript
- 运行形态：长期运行的权威服务端
- 架构：模块化单体
- 客户端通信：HTTP + WebSocket
- 持久化：PostgreSQL
- 首发客户端：Web 优先

### 暂定、可在搭建阶段调整的实现选择

- Fastify 作为 HTTP/WebSocket 接入层
- Zod 或等价方案校验网络协议与内容数据
- Vitest 作为规则与集成测试工具
- npm workspaces 与单一 lockfile 管理依赖与工作区

这些库不是架构承诺。替换它们不应改变模块边界、权威状态模型或一致性规则。

本 ADR 只冻结工程方向，不定义玩法。北郡 v1 的全部玩法语义以[北郡 v1 可执行设计包](../northshire-v1/README.md)为准，不从通用战斗设计隐式继承规则。

## 选择理由

### 开发速度优先于尚未出现的性能优化

当前核心难点是任务图、职业规则、内容生产和多人状态一致性，而不是计算吞吐。TypeScript 能覆盖服务端、协议定义、内容工具和 Web 客户端，减少跨语言模型同步成本。

### 单一权威进程简化多人一致性

房间、仇恨表、战利品归属和副本阶段都需要明确的事件顺序。在一个权威进程内串行处理相关命令，比过早引入跨服务消息、分布式锁和最终一致性更容易验证。

### 模块化保留未来拆分空间

模块之间通过明确的应用接口协作，禁止绕过模块直接修改其数据。若未来确有扩展需求，可以逐步提取接入、聊天、AI 或世界分区，而不需要现在承担分布式系统成本。

## 未选择的方案

### Rust 作为首版核心

暂不采用。当前没有性能分析、安全边界或单文件部署需求证明其收益高于双语言开发、序列化和调试成本。Rust 可在出现经过测量的性能热点或确定采用离线 Tauri 本地核心时重新评估。

### Tauri 作为首版基础架构

暂不采用。Tauri 是桌面客户端外壳，不能替代多人权威服务端。如果以后需要桌面安装包，可以复用 Web 客户端并增加薄桌面适配层。

### 微服务

暂不采用。它会让交易、组队、战斗和任务状态提前面对跨服务事务、消息排序、重试和故障恢复问题，而当前没有独立扩缩容或独立发布的需求。

### Evennia/Python

不作为当前主路线。Evennia 能快速提供传统 MUD 的账号、命令、持久化和 Telnet 基础设施，但项目现阶段更重视自定义连续自动战斗与低频决策、现代 Web 体验、严格内容 Schema 与前后端协议复用，因此选择 TypeScript 自研。

## 结果与约束

正面结果：

- 一套语言覆盖主要工程面；
- 多玩家状态只有一个权威来源；
- 首版部署、调试和本地开发较简单；
- 可以先验证玩法，再根据证据扩展架构。

需要承担：

- 自行实现账号、会话、命令处理和 MUD 领域能力；
- 单进程故障会影响所有在线玩家；
- 阻塞型任务必须移出实时命令路径；
- 必须持续维护清晰模块边界，避免演变为无边界的大型单体。

## 重新评估触发条件

只有出现至少一项可观测证据时才考虑拆分服务或引入 Rust：

- 单进程的 CPU、内存或事件循环延迟持续超出目标；
- 在线连接或世界规模要求水平扩展；
- 某一模块需要明显不同的资源模型或发布节奏；
- 需要跨区域部署或世界分片；
- 聊天、邮件等非核心模块的故障确实影响核心战斗，需要独立故障隔离；
- 监控已证明数据库事务或锁竞争是瓶颈，且领域边界允许明确的数据所有权；
- 桌面离线运行成为首发级需求；
- 性能分析定位出稳定、值得跨语言实现的热点。
