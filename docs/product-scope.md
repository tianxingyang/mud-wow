# 产品范围：Vanilla 人类线文字多人 RPG

> 文档状态：首个垂直切片范围基线  
> 更新日期：2026-07-15  
> 依据：[原始《魔兽世界》人类联盟线：文字 MUD 调研与 MVP 蓝图](../vanilla-wow-text-mud-research.md)、[Vanilla 人类联盟 1–31：任务链与区域路线核对表](../vanilla-wow-human-route-reference.md)、[战斗系统设计规范](../vanilla-wow-text-mud-combat-design.md)及当前项目讨论

## 1. 本文如何表达结论

本文使用三种状态，避免把研究建议误写成项目承诺：

- **已决定**：当前阶段按此执行，变更时应同步修改本文。
- **待决定**：实现前仍需做产品或规则选择，本文不预设答案。
- **后续建议**：研究材料给出的合理演进方向，不属于首个垂直切片承诺。

## 2. 产品定义

### 2.1 已决定

本项目是一款以 **Vanilla 1.12** 为资料基线、从普通人类冒险者视角体验暴风王国早期地方危机的**共享世界文字多人 RPG（MUD）**。

玩家通过文字观察环境、移动、与 NPC 交谈、推进任务和参与战斗。多个玩家连接同一个权威世界，可以在同一房间中看到彼此及相关世界事件。服务端统一裁定位置、任务、战斗、掉落与持久化状态，客户端只提交意图和显示结果。

当前产品目标不是复刻完整的 1.12 服务器，也不是一次完成 1–60 级世界。第一步只验证：原版北郡的地理、任务和职业战斗被转换成文字交互后是否仍然清楚、有探索感，并能在多人共享状态下稳定运行。

### 2.2 已决定的内容口径

- 叙事以 2004 年原版手册与旧版任务事实为参考。
- 任务、物品和世界关系以 Vanilla 1.12／Classic 旧世界资料为主要冻结点。
- 排除大灾变重做、探索赛季、周年服与现代正式服新增机制。
- 原版事实、文字版改编和项目新增内容必须明确区分。
- 不把多个独立地区危机强行改写成一条现代式统一主线。

### 2.3 当前不是产品目标

- 完整复刻《魔兽世界》客户端、地图或全部数值细节。
- 首发覆盖完整 1–30 或 1–60 内容。
- 以大型商业 MMORPG 的并发规模作为首阶段目标。
- 用生成式 AI 决定剧情事实、任务完成、伤害、掉落或永久世界状态。

## 3. 目标用户

### 3.1 已决定

首阶段服务于：

- 项目创建者及少量朋友；
- 对 Vanilla 人类早期任务线和暴风王国地方故事感兴趣的玩家；
- 能接受以阅读、地标判断和简洁命令为主要交互方式的玩家；
- 希望与其他真实玩家处在同一世界，而不只是游玩一份完全独立的单机存档的玩家。

### 3.2 待决定

- 是否面向不了解《魔兽世界》的新玩家提供额外背景引导。
- 是否在后续公开招募更大范围的测试者。
- 最终是否公开发布，以及是否商业化。

## 4. 设计原则

### 4.1 先证明一个闭环

先完成北郡 1–5 垂直切片，再扩展艾尔文森林、西部荒野或副本。新增系统必须直接服务于当前可玩的闭环，不为遥远需求预建通用平台。

### 4.2 文字地标承担导航

玩家应能依靠道路、山谷、矿洞、葡萄园、建筑和人物关系完成移动与任务。没有现代地图箭头不等于故意含糊；`look`、`exits` 和任务文本必须能重复提供关键线索。

### 4.3 服务端是唯一规则权威

多人共享世界中的位置、敌人、任务、战斗和奖励均由服务端裁定。WebSocket 连接层只传递命令与结果，不能直接修改世界状态。角色的任务进度和剧情知识按角色保存，不因其他玩家已完成内容而自动解锁。

### 4.4 规则结构化，AI 只做表现

任务前置、目标、奖励、伤害、掉落和世界变化由可测试的结构化数据与规则控制。AI 若后续启用，只能在已知事实内调整表达，不得新增重大人物关系、提前泄露剧情或越权改变状态。

### 4.5 保留“魔兽味”，裁掉低收益细节

优先保留职业资源差异、读条、打断、仇恨、品质、掉落、训练和任务链。精确偏斜、碾压、擦过、旧式法术批处理等不直接提升首切片体验的细节不进入当前范围。

战斗内部连续调度，但玩家通过持久战术自动执行普通动作，约每 5 秒收到一次决策摘要，并可覆盖下一合法动作。普通同级战斗应允许短暂离开；当前遭遇结束后停止，不自动攻击下一批敌人。战斗距离只使用接战、接近和脱离关系，不引入码数、地形、战斗区域或视线。

### 4.6 多人不等于微服务

不同玩家之间的交互通过同一个权威服务端完成。当前采用 TypeScript 模块化单体，让会话、世界、任务、战斗和持久化在一个部署单元中协作；只有出现经验证的扩展问题后才考虑拆分。

### 4.7 改编必须可追溯

每条任务和关键内容记录至少保留 `source_version`、`source_url`、`canonical_status` 与 `adaptation_notes`。内容事实可核对，改编边界可审查。

## 5. 首个垂直切片：北郡 1–5

### 5.1 切片目标

用一次约 **60–90 分钟**的完整流程验证以下假设：

1. 玩家不查外部攻略，也能凭文字地标从北郡修道院走到闪金镇。
2. 任务、战斗、掉落、升级、训练和存档能形成最小成长循环。
3. 战士与法师在资源和行动节奏上有明显区别。
4. 至少两个玩家可同时连接，并获得一致的房间存在与事件结果。
5. 退出重连后，关键角色状态能准确恢复。

### 5.2 已决定的内容范围

| 类别 | 首切片范围 |
|---|---|
| 地区 | 北郡修道院／北郡山谷，等级 1–5，出口止于前往闪金镇 |
| 房间 | 8–12 个，以修道院庭院、道路、回音山谷／矿洞、葡萄园／仓房、盗贼营地和南路等可辨识地标构成 |
| 任务 | 8–10 条 Vanilla 旧版任务改编 |
| 主干 | `A Threat Within`（783）→ `Kobold Camp Cleanup`（7）→ `Investigate Echo Ridge`（15）→ `Skirmish at Echo Ridge`（21）→ `Report to Goldshire`（54） |
| 支线候选 | 狼皮、盗贼兄弟会、加瑞克悬赏、米莉葡萄园任务；最终从已核对的旧版北郡支线中选取 |
| 敌人 | 4–6 类普通敌人，1–2 个命名敌人 |
| 职业 | 战士、法师；每职业先做 4–5 个关键技能 |
| 基础玩法 | 观察、出口与移动、NPC／任务交互、基础战斗、掉落、经验与升级、训练、背包／装备的最小闭环 |
| 多人验证 | 至少两个客户端同时连接；同一房间的玩家出现、离开及相关动作结果能同步 |
| 持久化 | 角色位置、生命、背包、装备及任务状态在退出重连后恢复 |

北郡内容必须遵守以下事实边界：

- 不使用大灾变后的兽人入侵任务。
- 北郡威胁是狗头人侵入回音山矿洞、迪菲亚占据葡萄园，以及地方狼群等问题。
- 不新增“狼群受人控制”“失踪农民被地下组织抓走”或黑铁矮人活动等未经原任务支持的剧情。

### 5.3 架构约束与当前实现候选

这些约束来自当前技术讨论，是首切片的实现边界，而非要求一次做完所有未来模块：

已经决定：

- TypeScript 模块化单体、单一权威服务端。
- 多个客户端连接同一个服务端，客户端不直接裁决游戏状态。
- 使用关系型数据库保存持久状态，当前首选 PostgreSQL。
- 首版不引入 Rust、Tauri、Redis、微服务或事件总线。

当前实现候选：

- Node.js 作为 TypeScript 运行时；
- Fastify + WebSocket 作为服务端接入基础；
- Zod 或等价方案用于协议和内容数据校验；
- Vitest 或等价方案用于规则测试。

候选库应在搭建阶段根据事务支持、运行时校验、可测试性和维护状态确认；更换具体库不改变本节的架构约束。

服务端内部可按 `session`、`world`、`quest`、`combat`、`social`、`persistence` 划分职责，但仍作为一个进程和一个部署单元运行。模块名称是代码边界，不代表独立服务。

### 5.4 明确不在首切片范围

- 艾尔文森林主体、暴风城、西部荒野、死亡矿井及其他地区。
- 盗贼、牧师、圣骑士和术士的可玩实现。
- 五人副本、完整组队系统和规则型队友控制器。
- 公会、邮件、拍卖行、专业技能及完整经济生态。
- 完整天赋树、武器熟练度还原和高等级系统。
- 赤脊山、暮色森林、部落出生线及 1–60 世界。
- 桌面安装包、Tauri 客户端和 Rust 游戏核心。
- 大规模水平扩展、跨服或服务拆分。
- 自由生成且能改变规则或剧情事实的 AI NPC。
- 对外公开发行所需的品牌、运营、付费与合规体系。

“不在首切片范围”只表示本阶段不做，不自动表示以后一定会做。

### 5.5 待决定事项

下列问题不阻塞服务端方向，但应在相应功能实现前形成明确规则：

- 首切片由最小 Web 客户端还是开发期文字界面承载；Web UI 框架尚未决定。
- 8–10 条任务中具体选择哪些支线，以及是否全部按原等级和奖励比例实现。
- 战士与法师的首批技能清单、伤害公式和具体数值。
- 多名玩家攻击同一敌人时的任务计数、经验和掉落归属。
- 战斗边界已经确定为 WorldInstance 内可独立存在的 CombatSession；敌人实例所有权、共享击杀、刷新和掉落归属规则仍待决定。
- 首切片是否需要 `say` 等主动社交命令；当前最低多人验收只要求存在与事件同步。
- 角色创建细节、命名规则和同账号角色数量。
- 死亡后的完整跑尸流程是否进入首切片，还是在后续阶段补齐。
- AI 表现层是否在首切片启用；其规则权限边界已经决定。

## 6. 核心用户流程

### 6.1 进入或恢复角色

1. 玩家连接服务端并进入已有角色。
2. 服务端读取权威存档，将玩家放回保存的位置。
3. 玩家看到房间描述、明显地标、出口、NPC、敌人与同房间玩家。
4. 断线或退出后再次进入，位置和关键进度保持一致。

角色首次创建的具体界面与规则属于待决定事项。

### 6.2 探索与移动

1. 玩家使用 `look` 查看当前环境和地标。
2. 使用 `exits` 确认可用方向。
3. 使用 `go` 前往相邻房间。
4. 房间描述随位置及已发生的结构化状态变化。
5. 玩家仅凭这些线索逐步从修道院、道路到达回音山、葡萄园和南路。

### 6.3 接取并完成任务

1. 玩家与 NPC 交谈并获得可理解的目标与方位线索。
2. 服务端检查任务前置与角色知识状态。
3. 玩家通过移动、调查、击败敌人或收集物品推进目标。
4. 进度由服务端记录，客户端不能直接声明完成。
5. 玩家返回交付，获得奖励并解锁后续任务或知识状态。

首切片应至少完整覆盖“接取 → 推进 → 交付 → 奖励 → 解锁后续”的主干任务流程。

### 6.4 战斗与成长

1. 玩家选择目标并发起遭遇。
2. 服务端按持久战术连续处理普通攻击、读条、资源消耗、伤害和敌人行动；玩家可以覆盖下一合法动作。
3. 客户端约每 5 秒收到压缩决策摘要，关键意图立即提示，不连续刷出无决策价值的日志。
4. 击败敌人后，服务端裁定任务进度、经验和掉落。
5. 玩家升级、装备所得物品，并在训练师处获得成长。

### 6.5 多人共享房间

1. 玩家 A 与玩家 B 同时连接服务端。
2. 一名玩家进入或离开房间时，另一名玩家看到一致的存在变化。
3. 同房间内由权威世界产生的相关动作结果对参与客户端保持一致。
4. 两名玩家各自的任务与知识状态独立保存。

共享击杀、奖励归属和正式组队流程仍属待决定事项，不在本流程中预设。

## 7. 可验证的验收标准

### 7.1 内容与路线

- [ ] 可玩区域包含 8–12 个互相连通、地标可辨识的北郡房间。
- [ ] 主干五个任务按正确前置顺序接取与完成，不能跳过前置直接解锁后续。
- [ ] 总计录入 8–10 条经核对的 Vanilla 北郡任务改编。
- [ ] 一次完整游玩测试中，测试者不借助外部地图或攻略，能根据房间、任务和 `look` 提示从修道院到达闪金镇出口。
- [ ] 完整流程目标时长处于约 60–90 分钟；若明显偏离，应记录原因并调整内容或节奏。
- [ ] 游戏中不出现大灾变兽人入侵或北郡事实边界之外的剧情。

### 7.2 任务与知识状态

- [ ] 至少一条任务自动化测试覆盖接取、目标推进、交付、奖励和后续解锁全过程。
- [ ] 无前置、重复交付、客户端伪造完成等无效操作不会改变权威状态。
- [ ] 每条录入任务具备来源版本、来源链接、内容性质和改编说明。
- [ ] NPC 不会向尚未取得相应知识状态的角色提前泄露后续事实。

### 7.3 战斗与成长

- [ ] 战士和法师各有 4–5 个可用关键技能，并能独立完成首切片普通内容。
- [ ] 两职业至少在资源方式、攻击节奏或读条行为上呈现可观察差异，而不是共用一套仅改名的技能。
- [ ] 服务端统一处理目标、资源、伤害、敌人行动、胜负、经验和掉落。
- [ ] 自动化测试覆盖至少一个完整战斗、一项资源／读条差异及一次胜利后的奖励结算。
- [ ] 使用基准装备和推荐战术发起同级普通战斗后停止输入两分钟，战斗能够结束并停止，不自动攻击下一只敌人。
- [ ] 默认战术不使用药水、稀有消耗品或长冷却；返回时只显示压缩摘要。
- [ ] 持久战术和一次性覆盖动作均有自动化测试，覆盖命令失效时能回落到合法动作。
- [ ] 近战不能命中未接战目标；普通接近需要时间，且首切片不存在码数、战斗区域或视线依赖。
- [ ] 升级、装备与训练造成的状态变化可以保存并在重连后恢复。

### 7.4 多人与一致性

- [ ] 两个独立客户端能够同时连接同一个服务端并进入同一房间。
- [ ] 玩家进入、离开或断线时，另一客户端看到的房间成员状态与服务端一致。
- [ ] 同一权威事件不会因客户端重复提交而被重复结算。
- [ ] 两名玩家的任务、背包和知识进度互不覆盖。
- [ ] 多人并发场景至少有一组自动化或集成测试，验证房间成员状态与角色数据隔离。

### 7.5 存档与恢复

- [ ] 非战斗退出后，任务状态、背包、装备、生命和位置准确恢复；战斗中断线则恢复服务端继续演算后的当前权威状态与战斗摘要，不回滚到断线瞬间。
- [ ] 玩家升级或获得物品后退出并重连，成长结果不丢失且不重复发放。
- [ ] 持久化失败时，服务端不能向客户端报告一次实际未提交成功的永久奖励。

### 7.6 切片完成判定

只有当以上验收项通过，并完成一次从进入北郡到到达闪金镇出口的真实游玩测试，首个垂直切片才视为完成。代码模块存在、接口可调用或单独的数据录入均不等同于切片完成。

后续范围按“先通过前一阶段验收，再确认下一阶段”的方式推进。

阶段 B 的规模估算、职业技能数量、专业技能与副本细节来自研究蓝图，尚未全部转化为产品承诺。正式启动阶段 B 前应另行冻结范围。

## 9. 技术方向的升级触发条件

### 9.1 已决定

首切片坚持 TypeScript 模块化单体。多人交互本身不是拆分服务的理由；同一权威进程更容易保证房间、任务、战斗和奖励的一致性。

### 9.2 以后再评估

只有出现实际证据时才重新评估：

- 单进程已经无法承担真实连接或战斗负载；
- 世界必须分区、多实例或多机器水平扩展；
- 某个模块出现独立发布或独立扩缩容的持续需求；
- 性能分析确认存在适合 Rust 处理的明确热点；
- 离线桌面安装包成为刚性产品目标，因而需要重新评估 Tauri。

这些触发条件不是当前路线图，也不意味着届时一定采用微服务、Rust 或 Tauri。

## 10. 版权与发布边界

### 10.1 当前边界

当前阶段定位为私人研究和玩法原型。项目可以研究旧版任务关系、事实和系统设计，但应自行改写文字表现，并保留来源与改编记录。

不得把以下内容直接作为项目发行资产：

- 暴雪任务原文或大段对话原文；
- 官方图标、界面、美术、模型、地图、音乐和音效；
- 从客户端或数据库直接导出的完整受保护数据资产；
- 容易让用户误认为官方产品的标志、名称组合或品牌包装。

`canonical_status: original` 只表示该条内容事实来自原版口径，不表示项目拥有原始文本或资产的授权。

### 10.2 公开发布前必须另行决定

- 产品最终名称、商标和视觉品牌。
- 是否替换或进一步抽象所有受保护的人名、地名、剧情和规则表达。
- 发布范围、许可证、分发方式及商业化模式。
- 是否需要专业知识产权与商标法律意见。

在这些事项完成独立审查之前，本文不把“公开发布”视为已获批准的项目阶段。本节仅用于界定工程与内容风险，不构成法律意见。

## 11. 当前决策摘要

### 已决定

- 做 Vanilla 1.12 口径的人类线共享世界文字多人 RPG。
- 先完成北郡 1–5 垂直切片，不铺开完整世界。
- 首切片先做战士和法师，验证任务、战斗、成长、多人同步和存档。
- 战斗采用连续自动执行、低频决策摘要和接战关系；规则细节以战斗系统设计规范为准。
- 使用 TypeScript 模块化单体权威服务端，关系型数据库当前首选 PostgreSQL。
- Node.js、Fastify、WebSocket、Zod 和 Vitest 是当前实现候选，不是不可替换的架构承诺。
- 首版不使用 Rust、Tauri、Redis 或微服务。
- 规则与剧情事实结构化，AI 只允许处于表现层。

### 待决定

- 首切片客户端形态与 Web UI 框架。
- 北郡支线、职业技能和战斗数值的最终清单。
- 多人共享击杀、任务计数、经验、掉落与敌人实例规则。
- 主动社交命令、角色创建和死亡流程进入首切片的程度。
- AI 表现层是否在首切片启用。
- 后续阶段的正式范围，以及项目是否公开发布。
