# 产品范围：Vanilla 人类线文字多人 RPG

> 文档状态：首个垂直切片范围基线  
> 更新日期：2026-07-17<br>
> 依据：[原始《魔兽世界》人类联盟线：文字 MUD 调研与 MVP 蓝图](research/vanilla-wow-text-mud-research.md)、[Vanilla 人类联盟 1–31：任务链与区域路线核对表](research/vanilla-wow-human-route-reference.md)、[战斗系统设计规范](combat-system.md)及当前项目讨论

北郡 v1 的全部玩法语义与验收口径以[北郡 v1 可执行设计包](northshire-v1/README.md)为唯一权威；本文件只负责更高层产品范围。通用战斗设计和其他通用文档不会被 v1 隐式继承，只有设计包明确引入的内容才进入 v1。

## 1. 本文如何表达结论

本文使用三种状态，避免把研究建议误写成项目承诺：

- **已决定**：当前阶段按此执行，变更时应同步修改本文。
- **待决定**：进入对应产品阶段前仍需做选择，本文不预设答案；是否阻塞当前 v1 由所在小节明确说明。
- **后续建议**：研究材料给出的合理演进方向，不属于首个垂直切片承诺。

## 2. 产品定义

### 2.1 已决定

本项目是一款以 **Vanilla 1.12** 为资料基线、从普通人类冒险者视角体验暴风王国早期地方危机的**共享世界文字多人 RPG（MUD）**。

玩家通过文字观察环境、移动、与 NPC 交谈、推进任务和参与战斗。多个玩家连接同一个权威世界，可以在同一房间中看到彼此及相关世界事件。服务端统一裁定位置、任务、战斗、掉落与持久化状态，客户端只提交意图和显示结果。

当前产品目标不是复刻完整的 1.12 服务器，也不是一次完成 1–60 级世界。第一步只验证：原版北郡的地理、任务和职业战斗被转换成文字交互后是否仍然清楚、有探索感，并能在多人共享状态下稳定运行。

### 2.2 已决定的内容口径

- 叙事以 2004 年原版手册与旧版任务事实为参考。
- 任务、物品和世界关系以 Vanilla 1.12／Classic 旧世界资料为主要冻结点。
- 排除大灾变重做、探索赛季、周年服与现代正式服新增机制。
- 原版事实、文字版改编和项目新增内容必须明确区分。
- 不把多个独立地区危机强行改写成一条现代式统一主线。

### 2.3 当前不是产品目标

- 完整复刻《魔兽世界》客户端、地图或全部数值细节。
- 首发覆盖完整 1–30 或 1–60 内容。
- 以大型商业 MMORPG 的并发规模作为首阶段目标。
- 用生成式 AI 决定剧情事实、任务完成、伤害、掉落或永久世界状态。

## 3. 目标用户

### 3.1 已决定

首阶段服务于：

- 项目创建者及少量朋友；
- 对 Vanilla 人类早期任务线和暴风王国地方故事感兴趣的玩家；
- 能接受以阅读、地标判断和简洁命令为主要交互方式的玩家；
- 希望与其他真实玩家处在同一世界，而不只是游玩一份完全独立的单机存档的玩家。

### 3.2 后续产品决定（不阻塞北郡 v1）

- 在北郡 v1 验证完成、面向更广泛测试者前，是否为不了解《魔兽世界》的新玩家提供额外背景引导。
- 是否在后续公开招募更大范围的测试者。
- 最终是否公开发布，以及是否商业化。

## 4. 设计原则

### 4.1 先证明一个闭环

先完成北郡 1–5 垂直切片，再扩展艾尔文森林、西部荒野或副本。新增系统必须直接服务于当前可玩的闭环，不为遥远需求预建通用平台。

### 4.2 文字地标承担导航

玩家应能依靠道路、山谷、矿洞、葡萄园、建筑和人物关系完成移动与任务。没有现代地图箭头不等于故意含糊；`look`、`exits` 和任务文本必须能重复提供关键线索。

### 4.3 服务端是唯一规则权威

多人共享世界中的位置、敌人、任务、战斗和奖励均由服务端裁定。HTTP／WebSocket 接入层只传递命令与结果，不能直接修改世界状态。角色的任务进度和剧情知识按角色保存，不因其他玩家已完成内容而自动解锁。

### 4.4 规则结构化，AI 只做表现

任务前置、目标、奖励、伤害、掉落和世界变化由可测试的结构化数据与规则控制。AI 若后续启用，只能在已知事实内调整表达，不得新增重大人物关系、提前泄露剧情或越权改变状态。

### 4.5 保留“魔兽味”，裁掉低收益细节

优先保留职业资源差异、读条、打断、仇恨、品质、掉落、训练和任务链。精确偏斜、碾压、擦过、旧式法术批处理等不直接提升首切片体验的细节不进入当前范围。

战斗内部连续调度，但玩家通过持久战术自动执行普通动作，约每 5 秒收到一次决策摘要，并可覆盖下一合法动作。普通同级战斗应允许玩家短暂不输入或离开界面；北郡 v1 不实现战斗内 `retreat`，只保留 `flee` 退出遭遇。当前遭遇结束后停止，不自动攻击下一批敌人。不引入码数、地形、战斗区域或视线；北郡 v1 实际启用的接战与移动动作只由其战斗规格定义。

### 4.6 多人不等于微服务

不同玩家之间的交互通过同一个权威服务端完成。当前采用 TypeScript 模块化单体，让会话、世界、任务、战斗和持久化在一个部署单元中协作；只有出现经验证的扩展问题后才考虑拆分。

### 4.7 改编必须可追溯

每条任务和关键内容记录至少保留 `source_version`、`source_urls`（数组）、`canonical_status` 与 `adaptation_notes`（数组）。内容事实可核对，改编边界可审查；不得同时保留旧的单值来源字段或字符串版 `adaptation_notes` 形成第二套 Schema。

## 5. 首个垂直切片：北郡 1–5

### 5.1 切片目标

用一次约 **60–90 分钟**的完整流程验证北郡 1–5 的探索、成长、战斗、多人一致性与恢复。首切片必须证明的假设与验收结果见[切片契约 §2](northshire-v1/00-slice-contract.md)与[验收与试玩 §11](northshire-v1/07-acceptance-and-playtest.md)。

### 5.2 已决定的内容范围

| 类别 | 首切片范围 |
|---|---|
| 地区 | 北郡修道院／北郡山谷，等级 1–5，出口止于前往闪金镇 |
| 房间 | 11 个，以修道院大厅／庭院、道路、回音山谷／矿洞、葡萄园／仓房、加瑞克棚屋和南路等可辨识地标构成 |
| 任务 | 10 条 Vanilla 旧版任务改编；其中 9 条可在切片内交付，`Report to Goldshire` 保持进行中 |
| 主干 | `A Threat Within`（783）→ `Kobold Camp Cleanup`（7）→ `Investigate Echo Ridge`（15）→ `Skirmish at Echo Ridge`（21）→ `Report to Goldshire`（54） |
| 支线 | 盗贼兄弟会、加瑞克悬赏、米莉葡萄园任务；狼类仅作环境敌人，不录入狼皮任务 |
| 敌人 | 5 类普通敌人、1 个命名敌人 Garrick Padfoot |
| 职业 | 战士、法师；完成对应训练后，每职业在 5 级恰好有 5 个当前可用动作，技能等级替换记录另计 |
| 基础玩法 | 观察、出口与移动、NPC／任务交互、基础战斗、掉落、经验与升级、训练、背包／装备的最小闭环 |
| 多人验证 | 至少两个客户端同时连接；同一房间的玩家出现、离开及相关动作结果能同步 |
| 持久化 | 角色位置、生命、背包、装备及任务状态在退出重连后恢复 |

北郡内容必须遵守以下事实边界：

- 不使用大灾变后的兽人入侵任务。
- 北郡威胁是狗头人侵入回音山矿洞、迪菲亚占据葡萄园，以及地方狼群等问题。
- 不新增“狼群受人控制”“失踪农民被地下组织抓走”或黑铁矮人活动等未经原任务支持的剧情。

### 5.3 架构约束

首切片采用 Node.js + TypeScript 模块化单体权威服务端、PostgreSQL 持久化和 HTTP + WebSocket Web 客户端。首版不引入 Rust、Tauri、Redis、微服务或事件总线。

本文件只记录这些产品级约束；组件、模块、数据边界和候选库由[系统架构](architecture.md)与 [ADR-0001](adr/0001-typescript-modular-monolith.md)负责。

### 5.4 明确不在首切片范围

- 艾尔文森林主体、暴风城、西部荒野、死亡矿井及其他地区。
- 盗贼、牧师、圣骑士和术士的可玩实现。
- 五人副本、完整组队系统和规则型队友控制器。
- 交易、竞争拾取、共享掉落和跨 CombatSession 治疗／合并。
- 公会、邮件、拍卖行、专业技能及完整经济生态。
- 完整天赋树、武器熟练度还原和高等级系统。
- 赤脊山、暮色森林、部落出生线及 1–60 世界。
- 桌面安装包、Tauri 客户端和 Rust 游戏核心。
- 大规模水平扩展、跨服或服务拆分。
- 自由生成且能改变规则或剧情事实的 AI NPC。
- 对外公开发行所需的品牌、运营、付费与合规体系。

“不在首切片范围”只表示本阶段不做，不自动表示以后一定会做。

### 5.5 北郡 v1 已冻结的执行决定

[北郡 v1 设计包](northshire-v1/README.md)已经冻结首切片的内容、战斗、战术、多人恢复和验收规则；当前没有等待实现自行选择的玩法项。具体规则由 `00`–`06` 各自拥有，完整证据映射由 `07` 负责，本文件不再重复列举。

前端框架、固定文案和具体基础库等不改变玩家语义的实现细节仍可在架构边界内选择。任何未写入 v1 设计包的玩法都视为未进入 v1，不能从通用战斗设计或后续阶段方向中默认补入。

## 6. 核心体验循环

玩家进入或恢复角色后，通过文字地标探索北郡，与 NPC 交谈并推进任务；战斗由持久战术连续执行，玩家只在需要时作低频选择。任务、战斗、掉落、升级、装备和训练共同形成成长闭环，多名玩家共享可见世界事件但保留个人任务与奖励状态。退出、断线或死亡后，玩家能够从服务端权威状态继续。

具体命令、状态变化、失败路径和端到端场景统一由[完整玩家流程](northshire-v1/01-player-flow.md)拥有；战斗、战术和多人恢复分别由 `04`、`05`、`06` 规格负责。

## 8. 技术方向的升级触发条件

### 8.1 已决定

首切片坚持 TypeScript 模块化单体。多人交互本身不是拆分服务的理由；同一权威进程更容易保证房间、任务、战斗和奖励的一致性。

重新评估只在出现实际证据时发生，触发条件见 [ADR-0001 重新评估触发条件](adr/0001-typescript-modular-monolith.md#重新评估触发条件)。

## 9. 版权与发布边界

### 9.1 当前边界

当前阶段定位为私人研究和玩法原型。项目可以研究旧版任务关系、事实和系统设计，但应自行改写文字表现，并保留来源与改编记录。

不得把以下内容直接作为项目发行资产：

- 暴雪任务原文或大段对话原文；
- 官方图标、界面、美术、模型、地图、音乐和音效；
- 从客户端或数据库直接导出的完整受保护数据资产；
- 容易让用户误认为官方产品的标志、名称组合或品牌包装。

`canonical_status: original` 只表示该条内容事实来自原版口径，不表示项目拥有原始文本或资产的授权。

### 9.2 公开发布前必须另行决定

- 产品最终名称、商标和视觉品牌。
- 是否替换或进一步抽象所有受保护的人名、地名、剧情和规则表达。
- 发布范围、许可证、分发方式及商业化模式。
- 是否需要专业知识产权与商标法律意见。

在这些事项完成独立审查之前，本文不把“公开发布”视为已获批准的项目阶段。本节仅用于界定工程与内容风险，不构成法律意见。

## 10. 当前决策摘要

### 已决定

- 做 Vanilla 1.12 口径的人类线共享世界文字多人 RPG。
- 先完成北郡 1–5 垂直切片，不铺开完整世界。
- 首切片先做战士和法师，验证任务、战斗、成长、多人同步和存档。
- 战斗采用连续自动执行、低频决策摘要和接战关系；北郡 v1 的规则细节以其[战斗与成长规格](northshire-v1/04-combat-and-progression.md)和[多人及恢复规格](northshire-v1/06-multiplayer-and-recovery.md)为准，通用战斗设计只负责 v1 以外的方向。
- 北郡 v1 使用 Node.js + TypeScript 模块化单体权威服务端、PostgreSQL，以及 HTTP + WebSocket 接入。
- Fastify、WebSocket 插件、Zod 和 Vitest 等具体库是当前实现候选，不是不可替换的架构承诺。
- 首版不使用 Rust、Tauri、Redis 或微服务。
- 规则与剧情事实结构化，AI 只允许处于表现层。

### 当前冻结边界与非阻塞选择

- 北郡 v1 的六项补充战斗边界已全部冻结，具体以 [v1 战斗规格 §1.1](northshire-v1/04-combat-and-progression.md#11-已冻结的补充边界)和[验收 §5.5](northshire-v1/07-acceptance-and-playtest.md#55-已冻结边界用例)为准；当前不存在阻塞实现的玩法决定。
- 前端框架、命令别名、名称字符集和 `say` 是否随首个构建交付属于 00 §8 的非阻塞实现选择，不需要玩法冻结，也不得改变可执行设计包的玩家语义。
- 后续阶段的正式范围，以及项目是否公开发布，不属于当前北郡 v1 的实现前置。
